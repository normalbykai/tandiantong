# Git 提交严格版

## 工作流

1. 先执行 `git status --short`，确认工作区是否有修改。
2. 阅读项目规则：根目录 `AGENTS.md`；如涉及版本文档或架构，再读取 `docs/README.md`、`docs/versions/v1.1/文档索引.md` 和相关设计文档。
3. 用 `git diff --stat`、`git diff --name-only` 和按文件 `git diff -- <path>` 查看修改内容。对新增文件用 `Get-Content` 或等效命令阅读关键内容。
4. 判断修改是否属于一个可独立验证的功能单元：
   - 如果修改明显混杂多个目的，先向用户说明并建议拆分提交。
   - 如果存在修改前已经有的无关改动，不要擅自回退；只 stage 本次要提交的文件。
   - 如果发现真实密钥、密码、完整令牌、生产凭据、个人敏感信息或构建产物，停止提交并说明风险。
5. 严格版必须根据变更范围运行最小相关验证：
   - Java 后端代码或配置变更：优先运行相关 Maven 模块测试；无法判断时运行 `mvn -pl tandiantong-bootstrap -am test`。
   - 前端代码变更：运行对应项目已有的类型检查、构建或测试命令。
   - 纯文档变更：可以不跑编译测试，但要说明“纯文档变更，未运行代码测试”。
   - 验证失败时不要提交，除非用户明确要求提交失败状态；默认先修复或报告。
6. 使用显式路径 stage 文件，避免误收无关改动。stage 后执行 `git diff --cached --stat` 和 `git diff --cached --check`。
7. 生成并执行本地提交。提交成功后报告提交哈希、提交信息、主要变更和验证结果。

## 提交信息规范

提交标题必须使用简体中文描述业务目的，允许使用英文类型前缀：

```text
type: 中文主题
```

允许类型：

```text
feat, fix, docs, refactor, test, build, chore
```

规则：

- 冒号后的主题必须是中文，禁止纯英文主题。
- 主题要描述业务目的或工程目的，禁止只写“更新”“修改”“临时提交”“WIP”。
- 技术名词、模块名、类名、版本号和缺陷编号可以保留英文。
- 单次提交应聚焦一个可理解的功能单元。
- 需要正文时，正文使用简体中文，说明变更原因、验证结果或注意事项。

示例：

```text
docs: 增加V1.1数据库基线规则
build: 调整Flyway基线迁移命名
fix: 修复旧库基线切换启动失败问题
feat: 增加退款重试记录表
```

## 命令约束

- 禁止使用 `git reset --hard`、`git checkout -- <path>` 或其他会丢失用户工作的命令，除非用户明确要求。
- 禁止用 `git add .`、`git add -A` 盲目 stage；必须基于已阅读的文件路径显式 stage。
- 不要 push，不要创建分支或 PR，除非用户另外明确要求。
- 如果用户只要求分析改动并生成提交信息，不需要提交，改用 `$git-commit-quick`。
- 如果提交成功，在最终回复中按 Codex 桌面规则包含本地提交结果；如工具要求 Git 指令，提交后附带对应 `::git-commit{...}` 指令。
