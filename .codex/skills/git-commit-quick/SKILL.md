# Git 提交快速版

## 工作流

1. 只执行 `git status --short`、`git diff --stat` 和 `git diff --name-only` 了解当前 Git 显示的改动。
2. 必须包含 `??` 未跟踪文件；Git 没显示的忽略文件、排除文件和构建产物不用管，不要额外扫描。
3. 对已跟踪修改，最多读取 3 个关键文件的 `git diff -- <path>`；优先选择能代表业务目的的文件。
4. 对未跟踪文件，优先根据文件路径和文件名判断用途；只有文件名不足以判断时，最多读取每个文件前 40 行，总计最多读取 3 个未跟踪文件。
5. 不做代码审查，不做安全扫描，不判断是否可编译，只回答“改了什么”和“完整提交信息建议”。
6. 根据已读取的改动提炼提交标题和正文，正文必须覆盖本次修改的主要范围、行为或配置变化。
7. 如果修改明显包含多个无关目的，给出多个完整提交信息建议；否则只给一个推荐提交信息。

## 禁止事项

- 禁止运行测试、编译、构建、格式化或静态检查。
- 禁止执行 `git add`、`git commit`、`git push`。
- 禁止修改文件。
- 禁止回退用户改动。
- 禁止扫描全仓库，禁止读取与当前 Git 修改无关的文件。
- 禁止分析 `.gitignore` 已排除且未出现在 `git status --short` 的文件。
- 禁止把快速版结果描述成已经验证通过。

## 提交信息规范

提交信息必须由标题和正文组成。标题使用简体中文描述业务目的，允许使用英文类型前缀：

```text
type: 中文主题
```

允许类型：

```text
feat, fix, docs, refactor, test, build, chore
```

选择建议：

- `feat`：新增业务功能或用户可感知能力。
- `fix`：修复缺陷、启动失败、错误行为或数据问题。
- `docs`：仅文档、规则、说明或 skill 文本变化。
- `build`：构建、依赖、迁移脚本、配置或工程基线变化。
- `test`：测试用例或测试基础设施变化。
- `refactor`：不改变行为的结构调整。
- `chore`：维护性杂项，优先少用。

规则：

- 冒号后的主题必须是中文，禁止纯英文主题。
- 主题要说明本次修改的目的，禁止只写“更新”“修改”“提交代码”。
- 标题只概括单一核心目的，不堆砌文件名和实现细节。
- 技术名词、模块名、类名、版本号和缺陷编号可以保留英文。
- 标题与正文之间必须保留一个空行。
- 正文使用 `- ` 开头的中文条目，通常写 2 至 5 条；每条描述一个具体改动，不重复标题。
- 正文应优先说明业务范围、结构调整、接口或配置变化，以及确实存在的测试改动。
- 只描述 Git 改动能够支持的事实，禁止猜测业务效果或写入未执行的验证结果。
- 输出的代码块必须可以直接作为提交标题和正文使用，不在代码块中加入“推荐提交信息”等标签。
- 快速版只输出建议，不执行提交。

## 输出格式

优先输出改动概括和一个包含标题、正文的完整提交信息：

改动概括：
- 规范接口 DTO 的存放位置和文档注解
- 增加 Knife4j 全局令牌配置

推荐提交信息（标题与正文）：

```text
refactor: 规范接口DTO并完善Knife4j文档配置

- 将Controller内部请求和响应对象迁移到独立DTO包
- 为接口字段补充中文描述和示例参数
- 增加Knife4j全局Authorization令牌配置
- 为业务枚举字段补充中文注释
```

如需要拆分，每个建议都必须包含各自的标题和正文：

改动概括：
- 建立 V1.1 数据库结构基线
- 增加项目级 Git 提交技能

建议拆分提交：

1. 数据库结构基线

```text
build: 建立V1.1数据库结构基线

- 增加V1.1阶段所需的核心业务表
- 补充必要的索引、约束和基础数据
```

2. Git 提交技能

```text
docs: 增加项目级Git提交技能

- 增加快速生成中文提交信息的项目技能
- 约束快速版只读取Git当前显示的改动
```

代码块之后必须明确说明未执行测试、暂存和提交，避免将快速分析误解为已验证或已提交。
