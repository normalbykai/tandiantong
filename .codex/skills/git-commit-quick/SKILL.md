# Git 提交快速版

## 工作流

1. 执行 `git status --short`，确认当前工作区修改。
2. 执行 `git diff --stat` 和 `git diff --name-only`，了解变更范围。
3. 对已修改文件读取关键 diff：
   - 使用 `git diff -- <path>` 查看已跟踪文件改动。
   - 对未跟踪新增文件，用 `Get-Content` 或等效命令阅读关键内容。
   - 只读取足够生成提交信息的内容，不做完整代码审查。
4. 判断这批修改的主要目的，并生成一个最合适的提交标题。
5. 如果修改明显包含多个无关目的，给出多个提交信息建议，并说明建议拆分。

## 禁止事项

- 禁止运行测试、编译、构建、格式化或静态检查。
- 禁止执行 `git add`、`git commit`、`git push`。
- 禁止修改文件。
- 禁止回退用户改动。
- 禁止把快速版结果描述成已经验证通过。

## 提交信息规范

提交标题必须使用简体中文描述业务目的，允许使用英文类型前缀：

```text
type: 中文主题
```

允许类型：

```text
feat, fix, docs, refactor, test, build, chore
```

选择建议：

- `feat`：新增业务功能或用户可感知能力。
- `fix`：修复缺陷、启动失败、错误行为或数据问题。
- `docs`：仅文档、规则、说明或 skill 文本变化。
- `build`：构建、依赖、迁移脚本、配置或工程基线变化。
- `test`：测试用例或测试基础设施变化。
- `refactor`：不改变行为的结构调整。
- `chore`：维护性杂项，优先少用。

规则：

- 冒号后的主题必须是中文，禁止纯英文主题。
- 主题要说明本次修改的目的，禁止只写“更新”“修改”“提交代码”。
- 技术名词、模块名、类名、版本号和缺陷编号可以保留英文。
- 快速版只输出建议，不执行提交。

## 输出格式

优先输出一个推荐提交信息：

```text
推荐提交信息：
build: 建立V1.1数据库结构基线
```

如需要拆分，输出多个建议：

```text
建议拆分提交：
1. build: 建立V1.1数据库结构基线
2. docs: 增加项目级Git提交技能
```
