# Git 本地提交快速版

## 模式判断

- 用户只要求“生成提交信息”“概括改动”“commit message”时，仅输出建议，不修改 Git 状态。
- 用户明确要求“提交”“本地提交”“快速提交”“一起提交”时，完成本地 `stage + commit`。
- 用户意图不明确时，仅输出建议，不擅自提交。

## 快速工作流

1. 执行 `git status --short`、`git diff --stat` 和 `git diff --name-only`，只处理 Git 当前显示的改动。
2. 必须包含 `??` 未跟踪文件；忽略 Git 未显示的排除文件、忽略文件和构建产物，不扫描全仓库。
3. 对已跟踪文件读取相关 `git diff -- <path>`；文件较多时优先读取能代表业务目的、配置、安全边界和测试变化的文件，不机械读取所有重复 DTO。
4. 对未跟踪文件先依据路径和文件名分类；需要确认内容时读取必要文件，禁止仅因文件多就遗漏准备提交的源码、测试或文档。
5. 将改动划分为一个或多个业务目的。存在明显无关改动时：
   - 仅生成建议：分别给出提交信息。
   - 要求本地提交：只提交用户明确指定的范围；无法判断范围时暂停并简短确认。
6. 选择提交类型并生成中文标题和正文。
7. 仅在本地提交模式下执行轻量提交前检查：
   - 执行 `git diff --check`。
   - 检查准备暂存的文件名和差异中是否明显包含密钥、令牌、密码、证书、构建产物或 IDE 私有配置。
   - 不运行测试、编译、构建、格式化、静态检查或全仓库安全扫描。
8. 本地提交模式下使用明确文件路径执行 `git add -- <paths>`，核对 `git diff --cached --stat` 和 `git status --short` 后执行 `git commit`。
9. 提交成功后返回提交哈希、完整标题、文件数量和剩余工作区状态；禁止 push。

## 安全边界

- 禁止提交与用户目标无关的文件。
- 禁止暂存 Git 未显示的文件、忽略文件、生成物、真实密钥和本地会话目录。
- 禁止使用 `git add -A`、`git add .` 或其他无法明确控制范围的暂存方式；必须列出路径。
- 禁止执行 `git push`、`git commit --amend`、`git reset --hard`、rebase 或历史重写。
- 不得把未执行的测试或编译描述为已通过；可以如实引用当前会话中已经执行且与改动对应的验证结果。
- `git diff --check` 或 `git commit` 失败时停止，报告错误，不通过跳过检查掩盖问题。
- 如果改动为空，直接说明无需提交。

## 提交信息规范

使用以下格式：

```text
type(scope): 中文主题

- 中文改动说明
- 中文改动说明
```

- 允许类型：`feat`、`fix`、`docs`、`refactor`、`test`、`build`、`chore`。
- `scope` 可省略；使用时应代表业务模块，例如 `api`、`order`、`security`。
- 冒号后的主题必须使用简体中文，简短说明核心业务目的，禁止只写“更新”“修改”或“提交代码”。
- 正文通常写 2 至 5 条，覆盖主要业务、接口、配置、测试或规则变化，不重复标题。
- 技术名词、模块名、类名、版本号和缺陷编号可以保留英文。
- 只描述 Git 差异能够支持的事实，不猜测效果。

类型选择：

- `feat`：新增业务功能或用户可感知能力。
- `fix`：修复缺陷、错误行为、兼容性或安全问题。
- `docs`：仅文档、规则或技能文本变化。
- `refactor`：结构调整且不改变业务行为。
- `test`：仅测试用例或测试基础设施变化。
- `build`：构建、依赖、迁移、配置或工程基线变化。
- `chore`：无法归入以上类型的维护性杂项，优先少用。

## 输出

仅生成建议时：

```text
改动概括：
- ...

推荐提交信息：

type(scope): 中文主题

- ...
- ...

未执行测试、暂存和提交。
```

本地提交成功时：

```text
已完成本地提交。

- 提交：<hash> <title>
- 改动：<数量或简要范围>
- 验证：未运行测试或编译（快速提交模式）
- 工作区：干净，或列出剩余未提交文件
```
