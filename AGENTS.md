# 摊点通项目开发规则

## 1. 适用范围与规则优先级

- 本文件适用于仓库内所有后端、管理后台、小程序、数据库脚本、测试、配置和文档。
- 开发前必须阅读 `docs/README.md`、`docs/versions/v1/README.md` 和 `docs/versions/v1/00-overview/architecture.md`，实现必须符合当前版本、当前阶段及已确认的业务边界。
- 新任务必须先确认所属版本和开发阶段，设计、计划、原型与验收文档必须放入对应版本目录，禁止将多个阶段持续堆叠在同一份文档中。
- 使用“必须”“禁止”“应当”“可以”表达约束强度。“必须”和“禁止”不可违反；偏离“应当”时必须说明理由。
- 若实现需求与现有设计或本规则冲突，必须先停止实现并确认、更新设计，禁止自行采用更宽松的解释。
- 子目录可以增加更具体的 `AGENTS.md`，但不得削弱本文件中的多租户隔离、交易安全、敏感信息保护和质量要求。

## 2. 开发原则与范围控制

- 第一阶段坚持 YAGNI，只实现已经确认且当前需要的能力。
- 禁止提前引入微服务、消息队列、搜索引擎、注册中心、配置中心、Kubernetes 等未确认基础设施。
- 禁止为多门店运营、套餐、人员排班、动态数据权限等未来能力创建空壳模块或无实际调用的抽象。
- 应当优先选择简单、清晰、可测试的实现；只有实际出现重复职责或扩展需求时才进行抽象。
- 修改必须聚焦当前任务，禁止夹带无关重构、批量格式化或无关依赖升级。

## 3. 中文注释规范

- 所有必要的代码注释、JavaDoc、复杂 SQL 注释和配置说明必须使用简体中文。
- 禁止新增英文解释性注释，禁止在同一条解释性注释中中英文混写。
- 类名、方法名、变量名、常量名、数据库对象、接口路径和标准技术标识必须使用规范英文命名，不得为了满足中文注释要求改用拼音命名。
- 业务规则、设计原因、状态流转、并发控制、安全边界、兼容处理和不直观算法必须添加中文注释。
- 简单赋值、普通条件判断、含义明确的方法和可直接从代码看出的逻辑不得添加重复性注释。
- 修改实现时必须同步检查和更新相关注释，禁止保留过期、错误或与代码矛盾的注释。
- 引用第三方协议的固定英文术语时，应在中文说明中保留必要术语，例如 `openId`、`access token`，但整句说明仍必须使用中文。

## 3.1 中文产品文档规范

- 原型、需求、产品设计、业务流程、验收说明和人工智能原型提示词的正文必须使用简体中文。
- 页面标题、菜单、按钮、字段名称、状态名称、错误提示、空状态文案和线框图标注必须使用中文。
- 面向人员阅读的文档文件名应当使用中文；约定文件名 `README.md`、`AGENTS.md` 可以保留英文，但正文仍必须使用中文。
- API 路径、代码标识、数据库字段、类名、标准协议和无法准确翻译的技术术语可以保留英文。
- 技术术语首次出现时应当提供中文含义，例如“基于角色的访问控制（RBAC）”。
- 人工智能原型提示词必须明确要求生成结果中的全部界面文案使用中文。
- 禁止生成纯英文页面、英文占位文案和无意义的中英文混排。
- 示例商户、商品、服务、人员、电话和地址必须使用自然、可理解的中文示例数据，敏感信息使用虚构或脱敏内容。

## 4. 工程与模块边界

- 后端必须采用 Java 21、Spring Boot 3 和 Maven 多模块的模块化单体架构。
- Maven 模块职责必须遵循架构设计，模块应当只承担一个清晰的业务职责。
- API 模块只负责协议适配、认证入口、参数校验和响应转换，禁止在 Controller 中编写核心业务规则。
- 业务模块之间必须通过明确的应用服务或接口协作，禁止跨模块直接访问对方的 Mapper、Repository 或数据库访问实现。
- `tandiantong-common` 只能保存稳定且真正通用的基础能力，禁止将业务模型、业务枚举或业务服务放入通用模块。
- 微信登录、微信支付、对象存储等第三方适配必须集中在 `tandiantong-integration`，业务模块禁止直接依赖第三方 SDK。
- 禁止形成循环 Maven 依赖；新增依赖必须说明用途，并优先复用项目已有能力。

## 5. Java 后端规范

- 包名必须全小写；类名使用大驼峰；方法、变量和字段使用小驼峰；常量使用全大写下划线形式。
- 单个类和方法应当保持职责单一。复杂业务流程应拆分为可命名、可测试的步骤，但禁止为了形式创建只有转发作用的无价值层级。
- Controller 不得直接返回数据库实体，必须使用明确的请求和响应对象。
- 输入参数必须在系统边界完成校验；业务不变量必须在应用服务或领域逻辑中再次校验。
- 业务异常必须使用统一错误码和中文可读消息，禁止向客户端暴露堆栈、SQL、第三方密钥或内部实现细节。
- 禁止使用 `double` 或 `float` 表示金额。系统金额必须使用整数“分”；需要金额对象时必须明确币种和单位。
- 时间必须明确时区和业务含义。营业日、支付时间、核销时间和预约时间不得混用无语义的字符串。
- 事务边界必须放在应用服务层，事务内禁止执行不可控的长时间网络调用。
- 关键状态变化必须使用明确状态机校验，禁止任意赋值跨越合法状态。

## 6. 前端规范

- C 端采用 uni-app 与 TypeScript，第一阶段只以微信小程序作为正式验收平台。
- B 端采用 Vue 3、TypeScript、Vite 和 Element Plus，并保证桌面管理与手机浏览器核销可用。
- 新增前端代码必须启用严格类型，禁止用无理由的 `any` 绕过类型检查。
- 页面组件负责展示与交互，业务请求、状态和可复用计算应当放在职责明确的模块中。
- 前端显示的菜单和按钮权限只用于改善体验，不得被视为安全边界；所有敏感操作必须由后端再次鉴权。
- 接口错误必须通过统一机制处理，禁止在各页面重复实现互相矛盾的登录失效和错误提示逻辑。
- 关键提交按钮必须防止重复点击，但后端仍必须保证幂等，不能依赖前端防重保障交易安全。

## 7. API、异常与日志

- REST API 必须遵守既定前缀：`/api/mini/v1/**`、`/api/admin/v1/**`、`/api/platform/v1/**` 和 `/api/callback/**`。
- API 变更必须保持请求、响应和错误码语义明确；破坏兼容性的变更必须升级版本或先获得确认。
- 接口必须返回统一响应结构和稳定业务错误码，禁止客户端依赖异常文本判断业务状态。
- 每个请求应当携带或生成追踪号，关键日志必须包含追踪号、业务标识和必要的租户标识。
- 日志内容必须使用中文描述业务事件；框架或第三方库自动输出的英文日志不受此限制。
- 禁止记录密码、验证码、完整访问令牌、刷新令牌、支付密钥、完整支付报文和其他敏感个人信息。
- 日志不得替代审计记录。退款、核销、角色变更、权限变更等关键操作必须写入专门的业务或审计记录。

## 8. 多租户与 RBAC 安全

- 所有租户业务表必须包含 `tenant_id`，门店相关数据必须同时包含 `store_id`。
- 业务请求中的 `tenant_id` 和 `store_id` 必须来自服务端可信上下文，禁止直接信任客户端提交值。
- MyBatis-Plus 多租户拦截器是基础隔离措施；订单、支付、退款、核销、角色和用户等敏感操作必须再次校验资源所属租户。
- 所有租户业务唯一索引必须包含 `tenant_id`，必要时同时包含 `store_id`。
- Redis 缓存键、分布式锁键、幂等键和限流键必须包含 `tenant_id`，禁止跨租户共享无隔离业务缓存。
- 定时任务和异步任务必须显式建立租户上下文，禁止在缺少租户范围时扫描并修改全部租户业务数据。
- 平台管理员与租户用户必须使用独立权限域，禁止通过伪造租户身份绕过权限模型。
- 菜单和按钮权限不能替代 API 鉴权；所有受保护接口必须由 Spring Security 和统一授权逻辑执行 RBAC 校验。
- 租户管理员只能授予自己拥有的权限，禁止越权创建权限标识或授予平台权限。
- 租户停用后必须阻止新交易和业务写操作，不得删除历史交易数据。

## 9. 商品、订单与预约规则

- 商品订单必须保存商品、SKU、加料项名称和成交价格快照，禁止查询实时商品数据还原历史订单。
- 库存变化必须记录库存流水，禁止只修改库存数字而不留下变更原因和业务来源。
- 创建待支付订单时锁定库存，支付成功后确认扣减，取消或支付超时后释放；退款成功后按已确认规则回补可售库存。
- 预约容量必须通过条件更新、版本控制或等价原子方式占用，禁止先查后改导致超额预约。
- 免费预约确认后生成凭证；付费预约必须在支付成功后生成凭证。
- 商品取餐号只用于识别，不能作为安全核销凭据；核销必须使用不可猜测的凭证或经后端验证的安全流程。
- 商品和预约核销必须执行带当前状态条件的原子更新，确保重复或并发核销最多成功一次。
- 已完成商品订单和已履约预约不得通过普通接口回退状态。

## 10. 支付、退款与幂等

- 微信支付回调必须验签，并通过平台业务单号反查订单和租户，禁止依赖登录会话或客户端租户参数。
- 创建订单、支付回调、退款请求、支付查询、定时任务和核销操作必须设计业务幂等键或等价幂等机制。
- 重复请求必须返回一致的业务结果，不得重复扣减库存、占用容量、退款或核销。
- 外部服务调用必须设置超时，并明确区分可重试和不可重试错误。
- 支付和退款状态必须以微信结果与本地合法状态机共同确认，禁止仅依据前端跳转结果修改交易状态。
- 第一阶段只支持核销前整单退款；部分退款和核销后在线退款不得在未更新设计前实现。
- 异步重试必须记录重试次数、下次执行时间、最终状态和最后错误，达到上限后必须可供人工排查。

## 11. 数据库、Redis 与迁移

- 数据库结构和基础数据变更必须通过 Flyway 脚本管理，禁止只手工修改本地或线上数据库。
- 已执行的 Flyway 脚本禁止修改；修正必须新增更高版本迁移。
- 表名和字段名使用小写下划线形式，避免使用数据库保留关键字；订单主表使用 `sales_order` 等明确名称。
- 表结构必须设置必要的主键、唯一约束、非空约束和索引，禁止仅依赖应用代码维护核心数据不变量。
- 订单、支付、退款、预约、库存流水、核销和审计记录禁止物理删除。
- 数据库查询必须显式考虑租户条件、数据量和索引；禁止在列表接口中引入无界查询或明显的 N+1 查询。
- Redis 只能用于缓存、短期锁、幂等和并发辅助，数据库仍是交易事实来源。
- 缓存更新必须有明确失效策略；禁止无限期缓存会影响商品、库存、权限或交易正确性的业务数据。

## 12. 测试与验证

- 新增或修改业务规则必须补充相应自动化测试；修复缺陷时必须先添加能够复现问题的测试。
- 金额计算、SKU 与加料组合、状态机、RBAC 和租户上下文必须有单元测试。
- 库存、预约容量、支付、退款和核销必须有模块集成测试，并覆盖重复请求和并发边界。
- 租户隔离测试必须至少使用两个租户，验证查询、更新、缓存和导出不会串租户。
- 第三方集成必须覆盖正常结果、验签失败、超时、重复回调和重试场景。
- 完成修改前必须运行与改动直接相关的最小测试集；提交前应当运行模块完整测试和静态检查。
- 禁止在测试失败、编译失败或静态检查失败时宣称任务完成。无法执行验证时必须明确说明未验证内容和原因。
- 禁止通过删除测试、降低断言、跳过检查或扩大忽略范围来掩盖失败。

## 13. 配置与敏感信息

- 密码、令牌、微信支付密钥、证书、数据库凭据和对象存储密钥禁止提交到仓库。
- 敏感配置必须通过环境变量、部署密钥或安全配置注入；示例配置只能使用明确的占位值。
- 上传文件必须校验类型、大小、扩展名和访问权限，禁止仅相信客户端提供的 MIME 类型。
- 对外响应、日志、异常和测试数据必须避免暴露真实个人信息和生产凭据。
- 新增第三方依赖前必须检查维护状态、许可证、已知安全风险和实际必要性。

## 14. Git 与交付规则

- 开始修改前必须检查工作区状态，保留并避开其他人员已有的未提交变更。
- 禁止使用 `git reset --hard`、未经授权的文件回退或其他可能丢失用户工作的操作。
- 开发任务应当先按业务阶段拆分，再将阶段内工作拆为可独立验证的功能单元。
- 每完成一个功能单元并通过相关验证后应当及时提交，禁止等整个项目或整个阶段全部完成后只创建一个巨大提交。
- 禁止按文件、方法或无业务意义的小改动机械拆分提交，避免产生大量无法独立理解和验证的碎片提交。
- 一个功能单元提交必须聚焦单一目的，并应当同时包含该功能必要的数据库迁移、实现、测试和相关文档。
- 每次功能提交必须能够正常编译，并通过与改动直接相关的测试和静态检查；禁止提交已知不可运行的半成品。
- 阶段完成后必须执行阶段范围内的整体测试和联调，由用户确认后再进入下一阶段。
- 默认在 `codex/` 前缀的功能分支中开发，阶段完成后由用户决定合并、创建 Pull Request 或保留分支。
- 人工创建的 Git 提交信息必须使用简体中文描述，仅约束后续提交，不重写已有提交历史。
- 允许使用 `feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`build:`、`chore:` 等英文类型前缀，但冒号后的主题必须使用中文。
- 功能、修复、重构、文档、测试、构建和配置等人工提交禁止使用纯英文主题。
- Git 自动生成的合并提交，以及拉取或同步代码产生的提交，可以保留默认英文信息。
- 技术名词、模块名、类名和缺陷编号可以保留英文，但提交信息的主要描述必须为中文。
- 禁止只写类型前缀、编号或无意义短语来规避中文描述要求。
- 合规示例：`feat: 增加商品库存锁定功能`、`fix: 修复支付回调重复扣减库存问题`。
- 禁止提交构建产物、IDE 私有配置、临时文件、真实密钥和本地可视化会话目录。
- 代码评审必须优先检查租户隔离、权限绕过、交易幂等、并发一致性、敏感信息和缺失测试。
- 任务完成说明必须列出主要变更、验证命令及结果，以及仍存在的明确限制。

## 15. 规则维护

- 本规则应当随已确认架构和实际工程实践演进，但任何修改都必须保持具体、可执行和可验证。
- 新增规则前必须确认它解决真实问题，禁止加入无法执行或只增加形式负担的要求。
- 业务范围或核心技术选型发生变化时，必须先更新架构设计，再同步更新本规则。
