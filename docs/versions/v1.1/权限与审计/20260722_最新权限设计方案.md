# V1.1 最新权限设计方案

## 1. 设计目标

平台管理端权限分为菜单视图权限和接口业务权限两层，解决菜单入口、页面访问和后端接口授权混用的问题。菜单权限用于改善操作体验，接口权限是后端最终安全边界。

本阶段先实施平台权限域，商户权限域沿用相同规则，待商户端功能进入实施阶段后接入。

## 2. 权限类型

### 2.1 菜单视图权限（VIEW）

`VIEW` 权限只表示用户可以看到并进入某个菜单或页面，不代表可以调用页面中的任何接口。

权限编码格式：

```text
{domain}:{menu或模块}:{资源}:view
```

平台端示例：

| 权限编码 | 用途 |
| --- | --- |
| `platform:merchant:view` | 查看租户管理入口 |
| `platform:operation-log:view` | 查看操作日志入口 |
| `platform:system:view` | 查看系统管理页面 |
| `platform:access:view` | 查看权限管理目录 |
| `platform:access:account:view` | 查看平台账号页面 |
| `platform:access:role:view` | 查看平台角色页面 |
| `platform:access:permission:view` | 查看平台权限页面 |

系统管理页面内部的“平台设置、字典管理、安全设置”属于功能分段，不是左侧菜单，不单独建立 `VIEW` 权限。进入系统管理页面后，具体分段和操作由对应的 `API` 权限控制。

### 2.2 接口业务权限（API）

`API` 权限用于后端接口和具体业务操作，由 Sa-Token 的 `@SaCheckPermission` 校验。

权限编码格式：

```text
{domain}:{resource}:{action}
```

常用动作含义：

| 动作 | 含义 |
| --- | --- |
| `read` | 查询接口 |
| `create` | 新增 |
| `update` | 修改 |
| `delete` | 删除 |
| `status:update` | 启用或停用 |
| `assign` | 分配或授权 |
| `reset` | 重置 |

`read` 不再承担菜单可见职责。页面需要显示时，必须同时配置对应的 `VIEW` 权限和实际需要的 `API` 权限。

## 3. 授权规则

角色只直接分配权限点，不直接绑定前端菜单。前端菜单通过 `VIEW` 权限过滤，按钮通过对应的 `API` 权限过滤，后端接口始终独立执行 `API` 权限校验。

| 场景 | 所需权限 |
| --- | --- |
| 显示菜单和进入页面 | 对应 `VIEW` |
| 查询页面数据 | 对应 `API` 的 `read` |
| 显示新增按钮并提交 | 对应 `API` 的 `create` |
| 显示编辑按钮并提交 | 对应 `API` 的 `update` |
| 显示启停按钮并提交 | 对应 `API` 的 `status:update` |

仅隐藏菜单、按钮不能构成安全边界。即使前端未显示入口，直接请求后端仍必须被接口权限拦截。

## 4. 平台权限归类

平台菜单与权限归类如下：

```text
工作台
└─ 平台权限域公共入口，不单独授权

租户管理
└─ platform:merchant:view
   └─ platform:merchant:read/create/enable/disable/invitation:reissue

操作日志
└─ platform:operation-log:view
   └─ platform:operation-log:read

系统管理
└─ platform:system:view
   ├─ 平台设置：platform:system:config:read/update
   ├─ 安全设置：platform:system:security:read/update
   └─ 字典管理：platform:dictionary:read/create/update/status:update

权限管理
├─ platform:access:account:view
│  └─ platform:account:read/create/update/status:update/password:reset
├─ platform:access:role:view
│  └─ platform:role:read/create/update/status:update/permission:assign
└─ platform:access:permission:view
   └─ platform:permission:read
```

## 5. 迁移与兼容策略

- 已执行的 API 权限编码不重命名、不删除，保证既有接口和角色关系兼容。
- 通过新的 Flyway 迁移增加 `VIEW` 权限及默认角色授权。
- 权限主键使用新的稳定编号，权限业务唯一性以 `domain + permission_code` 为准。
- 角色授权页面按 `VIEW` 和 `API` 两类展示，并明确说明两者用途。
- 后续新增左侧菜单页面必须同时评估页面 `VIEW` 权限和接口 `API` 权限；页面内部新增功能分段只新增对应的 `API` 权限，不新增菜单 `VIEW` 权限。

## 6. 验收标准

- 不同平台角色只能看到其被授予的菜单入口。
- 只有 `VIEW` 权限时可以进入页面，但没有接口 `read` 权限时后端查询仍被拒绝。
- 角色授权页面能区分菜单视图权限和接口业务权限。
- 平台超级管理员拥有全部平台 `VIEW` 和已有 `API` 权限。
- 权限变更不修改历史 Flyway 文件，且迁移审计记录版本、校验和与执行结果。
