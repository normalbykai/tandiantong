# V1.1 现行权限码规划与清单

## 1. 当前权限模型

平台权限分为两类：

```text
VIEW：控制左侧菜单和页面入口
API：控制后端接口和具体业务操作
```

角色只分配权限点，不直接绑定菜单。前端使用 `VIEW` 权限改善展示，后端使用 `API` 权限执行最终鉴权。

菜单隐藏、按钮隐藏和路由拦截都不能替代后端接口鉴权。

## 2. 权限编码格式

### 2.1 VIEW 菜单视图权限

格式：

```text
{权限域}:{菜单模块}:view
```

当前平台端示例：

```text
platform:merchant:view
platform:operation-log:view
platform:system:view
platform:access:account:view
platform:access:role:view
platform:access:permission:view
```

`VIEW` 只对应真实的左侧菜单或页面入口。系统管理页面内部的“平台设置、字典管理、安全设置”是页面分段，不单独建立 `VIEW` 权限。

### 2.2 API 接口业务权限

格式：

```text
{权限域}:{业务资源}:{动作}
```

常用动作：

| 动作 | 含义 |
| --- | --- |
| `read` | 查询 |
| `create` | 新增 |
| `update` | 修改 |
| `delete` | 删除 |
| `enable` | 启用 |
| `disable` | 停用 |
| `status:update` | 更新启停状态 |
| `assign` | 分配权限或角色 |
| `reset` | 重置 |
| `verify` | 核销或验证 |

`read` 只表示接口查询，不表示菜单可见。

## 3. 当前平台端菜单与权限

### 3.1 工作台

工作台是平台权限域的公共入口，当前不单独配置 `VIEW` 或 `API` 权限。

### 3.2 租户管理

菜单权限：

```text
platform:merchant:view
```

接口权限：

```text
platform:merchant:read
platform:merchant:create
platform:merchant:enable
platform:merchant:disable
platform:merchant:invitation:reissue
```

### 3.3 操作日志

菜单权限：

```text
platform:operation-log:view
```

接口权限：

```text
platform:operation-log:read
```

### 3.4 系统管理

系统管理是一个左侧菜单和页面，不拆分内部功能菜单。

菜单权限：

```text
platform:system:view
```

页面内部功能及接口权限：

| 页面分段 | API 权限 |
| --- | --- |
| 平台设置 | `platform:system:config:read`、`platform:system:config:update` |
| 安全设置 | `platform:system:security:read`、`platform:system:security:update` |
| 字典管理 | `platform:dictionary:read`、`platform:dictionary:create`、`platform:dictionary:update`、`platform:dictionary:status:update` |

页面分段不创建 `platform:system:config:view`、`platform:system:security:view` 或 `platform:system:dictionary:view`。

### 3.5 权限管理

权限管理是左侧目录，下面有三个真实页面菜单：平台账号、平台角色、平台权限。

平台账号：

```text
VIEW：platform:access:account:view
API：platform:account:read
     platform:account:create
     platform:account:update
     platform:account:status:update
     platform:account:password:reset
```

平台角色：

```text
VIEW：platform:access:role:view
API：platform:role:read
     platform:role:create
     platform:role:update
     platform:role:status:update
     platform:role:permission:assign
```

平台权限：

```text
VIEW：platform:access:permission:view
API：platform:permission:read
```

当前权限管理目录本身没有单独的 `platform:access:view` 数据权限。只要任意一个子页面 `VIEW` 权限存在，前端就展示权限管理目录。

## 4. 角色授权页面展示规则

角色授权页面按业务菜单模块归类：

```text
租户管理
操作日志
系统管理
  - 菜单视图权限
  - 接口查询权限
  - 接口业务权限
平台账号
平台角色
平台权限
```

其中：

- `platform:dictionary:*` 归入“系统管理”，不归入“其他权限”；
- `platform:system:config:*` 归入“系统管理”；
- `platform:system:security:*` 归入“系统管理”；
- 系统管理只有 `platform:system:view` 一个菜单视图权限；
- `VIEW` 和 `API` 是授权点类型，不是两套互相替代的安全机制。

## 5. 授权判断规则

| 操作 | 判断权限 |
| --- | --- |
| 显示左侧菜单 | 对应 `VIEW` |
| 进入页面 | 对应 `VIEW` |
| 查询页面数据 | 对应 API `read` |
| 显示新增并提交 | 对应 API `create` |
| 显示编辑并提交 | 对应 API `update` |
| 显示启停并提交 | 对应 API `enable`、`disable` 或 `status:update` |
| 配置角色权限 | `platform:role:permission:assign` |

后端 Controller 必须使用 `@SaCheckPermission` 校验 API 权限。前端权限判断只用于菜单、按钮和用户体验。

## 6. 命名约束

- 新增左侧菜单时，新增一个对应的 `VIEW` 权限。
- 新增已有页面内部功能分段时，只新增对应 `API` 权限，不新增 `VIEW` 权限。
- 同一业务资源的查询统一使用 `:read`，禁止用 `:view` 代替接口查询。
- `:view` 只用于菜单和页面入口。
- 权限码不得绑定具体前端 URL、按钮名称或数据库主键。
- 平台端使用 `platform` 域，商户端后续使用 `tenant` 域。
- 已执行权限码不直接重命名或删除，变更必须通过新的 Flyway 迁移兼容处理。

## 7. 当前数据库状态

当前已落地的平台 `VIEW` 权限为：

```text
platform:merchant:view
platform:operation-log:view
platform:system:view
platform:access:account:view
platform:access:role:view
platform:access:permission:view
```

平台 API 权限以数据库 `permission` 表和后端 `@SaCheckPermission` 使用的权限码为准。权限管理页面通过分页接口查询，默认每页最多返回 20 条；角色授权使用独立的全量权限选项接口。

## 8. 后续扩展示例

如果新增左侧“经营数据”菜单：

```text
VIEW：tenant:analytics:view
API：tenant:analytics:dashboard:read
     tenant:analytics:export:create
```

如果在系统管理页面内部新增“短信配置”分段：

```text
不新增 VIEW
API：platform:system:sms:read
     platform:system:sms:update
```

只有当“短信配置”未来变成独立左侧菜单时，才新增：

```text
platform:system:sms:view
```
