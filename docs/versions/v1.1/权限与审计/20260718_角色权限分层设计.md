# 角色权限分层设计

## 目标

将系统最高管理权、接口授权与前端交互控制分层，避免角色调整造成超级管理员永久失去权限，同时保持后端接口为最终安全边界。

## 权限层级

1. 普通角色授权：首席管理员、系统管理员和自定义角色统一按照实际分配的权限码执行，不因角色名称或 `is_authority_role` 标识自动补充权限。
2. 业务操作权限：编码示例 `platform:role:permission:assign`，由后端 `@SaCheckPermission` 校验，是安全边界；拥有该操作权限即可以配置平台角色权限，前端按钮和菜单复用同一权限码。
3. 按钮控制：不单独建立或授权按钮权限。按钮根据对应业务操作权限码控制显示和可操作性，但不能替代后端鉴权。
4. 菜单控制：菜单可以配置或映射到业务操作权限码，仅控制前端导航可见性；路由和接口不得仅依赖菜单控制。

## 按钮权限落地规则

- 一个业务动作只使用一个稳定权限编码，例如 `模块:资源:动作`；API 映射、按钮显示和菜单显示均引用该编码。
- 前端登录响应或会话查询必须获得当前权限编码集合，并通过统一的 `hasPermission` 判断按钮是否显示或禁用。
- 不得用按钮权限替代 API 鉴权。即使按钮被隐藏，直接构造请求仍会被后端 API 权限校验拦截。
- 第一阶段不为尚未实现的按钮预建空权限点；新增具体业务操作时同步新增权限定义、后端校验、前端控制和测试。
