# 角色权限分层设计

## 目标

将系统最高管理权、接口授权与前端交互控制分层，避免角色调整造成超级管理员永久失去权限，同时保持后端接口为最终安全边界。

## 权限层级

1. 受保护管理角色：平台“首席管理员”和“系统管理员”持有 `is_authority_role` 标识。持有该角色的账号可管理任意平台角色、角色权限和账号角色，且该能力不随角色的普通权限勾选变化而丢失。该标识不能通过普通角色编辑接口授予。
2. API 权限：编码示例 `platform:role:permission:assign`，由后端 `@SaCheckPermission` 校验，是安全边界。
3. 按钮权限：编码示例 `platform:role:permission:save`，只控制前端按钮显示和可操作性；按钮调用的 API 仍必须拥有对应 API 权限。
4. 菜单权限：编码示例 `platform:role:menu`，只控制前端导航可见性；路由和接口不得仅依赖菜单权限。

## 按钮权限落地规则

- 一个业务动作必须使用两个不同编码：`模块:资源:动作` 表示 API 权限，`模块:资源:动作:button` 表示按钮权限。
- 前端登录响应或会话查询必须获得当前权限编码集合，并通过统一的 `hasPermission` 判断按钮是否显示或禁用。
- 不得用按钮权限替代 API 鉴权。即使按钮被隐藏，直接构造请求仍会被后端 API 权限校验拦截。
- 第一阶段不为尚未实现的按钮预建空权限点；新增具体业务按钮时同步新增权限定义、后端校验、前端控制和测试。
