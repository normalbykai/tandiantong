# V1.1 项目完成度与后续阶段对齐

## 1. 对齐结论

当前 V1.1 已完成技术基线重构、认证权限基础、商品库存持久化、订单二期闭环补强的主要实现。按照 [后续开发路线](后续开发路线.md)，下一阶段不是继续扩展订单，而是进入第三阶段：预约完整闭环。

第三阶段的目标是把当前已有的“免费预约、容量占用、取消释放”基础能力，补齐为可支付、可回调确认、可超时释放、可核销履约、可由顾客查询和取消的完整预约交易链路。

## 2. 当前完成度

### 2.1 已完成或基本完成

| 范围 | 当前状态 | 说明 |
| --- | --- | --- |
| 技术基线 | 已完成 | 后端已统一为 Java 21、Spring Boot 3、MyBatis-Plus、Sa-Token、Knife4j 和 Maven 多模块 |
| 持久层重构 | 已完成 | 业务服务不再使用 `JdbcTemplate`，持久化访问集中在 Mapper |
| 认证基础 | 已完成 | 自研 JWT 已替换为 Sa-Token，平台与租户权限域分离 |
| 接口文档基础 | 已完成 | 后台与小程序接口已接入 Knife4j，并要求中文接口说明 |
| 商品与库存 | 已完成基础能力 | 商品、SKU、加料、库存流水已持久化，库存锁定、确认扣减和退款回补已有应用服务 |
| 第一阶段权限与审计 | 大部分完成 | 商品、退款、核销、经营数据和导出接口已有权限声明，关键退款与核销审计已补强；角色管理接口及双租户集成测试仍需继续补齐 |
| 第二阶段订单完整闭环 | 主体完成 | 加料报价、成交快照、待支付取消、超时取消、顾客订单查询与取消、退款失败重试和人工排查状态已完成主要实现 |
| 核销基础 | 已完成基础能力 | 核销凭证哈希保存、重复核销幂等、业务完成回调已有基础实现 |

### 2.2 当前已知限制

- 微信支付和退款仍是开发替身，第四阶段前不能视为真实微信支付接入完成。
- 预约模块已有免费预约和容量占用基础，但付费预约的预支付、回调确认、超时释放和凭证签发链路尚未完整闭合。
- 预约核销后的履约状态更新需要在第三阶段补齐为原子状态流转，并和核销模块完成可靠协作。
- 顾客侧预约列表、详情、取消接口仍未补齐，当前小程序侧主要是查询可预约服务和提交预约。
- 项目规则已明确新增或重构 Java 类型禁止使用 `record`，第三阶段修改预约模块时，应同步把被触达的预约命令、结果和值对象逐步转换为普通 `class`。

## 3. 第三阶段：预约完整闭环

### 3.1 阶段目标

第三阶段要完成预约从“创建”到“支付确认或免费确认”、再到“凭证签发、取消释放、核销履约、顾客查询”的完整闭环。阶段完成后，预约应达到和商品订单类似的交易安全水平：租户隔离、容量原子更新、支付回调幂等、状态机约束、凭证不可猜测、重复核销最多成功一次。

### 3.2 建议拆分

#### 功能单元一：预约支付与确认

- 为付费预约创建预支付单，返回预支付标识和支付参数。
- 支付回调通过预约单号反查可信租户和门店，禁止依赖客户端租户参数。
- 回调验签成功且金额、状态合法后，将预约从 `PENDING_PAYMENT` 原子更新为 `CONFIRMED`。
- 免费预约提交后直接确认，付费预约必须支付成功后确认。
- 支付成功后为预约签发核销凭证，明文令牌只在首次签发响应中返回。

#### 功能单元二：预约超时取消与容量释放

- 为待支付预约设置过期时间。
- 增加待支付预约超时取消能力。
- 超时取消必须原子更新预约状态，并释放已占用容量。
- 重复取消或并发取消必须返回一致结果，不得重复释放容量。

#### 功能单元三：预约核销与履约状态

- 核销模块识别预约业务类型后，通知预约模块完成履约。
- 预约模块必须使用带当前状态条件的原子更新，将 `CONFIRMED` 更新为 `FULFILLED`。
- 已取消、待支付或已履约预约不得再次履约。
- 核销记录、预约状态和凭证状态必须保持一致。

#### 功能单元四：顾客预约查询与取消

- 增加顾客预约列表接口。
- 增加顾客预约详情接口。
- 增加顾客在允许范围内取消预约的接口。
- 顾客侧查询和取消必须基于可信租户、门店和顾客身份或当前临时身份边界，不得仅靠客户端传入租户和门店。

#### 功能单元五：测试与验收

- 覆盖免费预约、付费预约、重复回调、错误金额、非法状态回调。
- 覆盖预约容量并发占用和释放。
- 覆盖超时取消与手动取消并发边界。
- 覆盖预约核销重复提交和跨租户核销。
- 至少使用两个租户验证预约查询、取消、核销和容量不会串租户。

### 3.3 第三阶段边界

- 不实现多门店运营、人员排班、套餐预约、动态数据权限和复杂服务资源调度。
- 不提前引入消息队列、定时任务平台或分布式事务组件；若需要定时扫描，应先使用当前模块化单体内的简单实现。
- 不接入真实微信支付证书和密钥细节；真实微信支付适配归第四阶段，但第三阶段应保留清晰接口和验签入口。
- 不做大范围无关重构；仅在触达预约相关类型时按规则逐步将 `record` 转为普通 `class`。

## 4. 后续阶段内容

### 第四阶段：外部服务

- 接入微信小程序登录并绑定顾客身份。
- 接入真实微信支付证书、密钥、回调验签、退款查询和错误分类。
- 按已确认需求接入对象存储和短信，所有敏感配置通过环境变量或安全配置注入。
- 覆盖超时、验签失败、重复回调、退款查询和重试场景。

### 第五阶段：联调与发布验收

- 完成双租户、库存并发、预约并发和重复核销集成测试。
- 完成小程序、商户后台、平台后台和后端接口的关键流程联调。
- 完成容器部署、生产配置、健康检查、数据库备份恢复验证。
- 汇总发布限制、回滚方案、未接入外部依赖和人工验收记录后，再形成正式版本提交。

## 5. 下一步建议

第三阶段开始前，应先补一份独立的第三阶段设计或实施清单，建议放在 `docs/versions/v1.1/预约闭环/` 目录下。该设计至少明确预约状态机、支付回调幂等键、超时取消策略、容量释放规则、凭证签发时机、顾客身份边界和测试用例清单。
