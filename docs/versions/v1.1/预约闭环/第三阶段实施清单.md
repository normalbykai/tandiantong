# 第三阶段预约完整闭环实施清单

## 1. 阶段目标

第三阶段围绕预约交易链路补齐付费确认、超时取消、核销履约、顾客查询与取消能力。阶段完成后，预约流程必须满足租户隔离、容量原子更新、支付回调幂等、状态机约束和凭证不可猜测等交易安全要求。

## 2. 功能单元拆分

### 2.1 预约支付与确认

- 付费预约提交后创建微信预支付单，保存预支付标识、支付金额和过期时间。
- 免费预约提交后直接进入 `CONFIRMED`，并签发预约核销凭证。
- 微信支付回调必须通过预约单号反查可信租户和门店，不依赖客户端租户参数。
- 回调验签成功且金额一致时，将 `PENDING_PAYMENT` 原子更新为 `CONFIRMED`。
- 重复回调返回已确认结果，不重复签发凭证。

### 2.2 预约超时取消与容量释放

- 待支付预约必须设置过期时间。
- 提供按租户门店扫描并取消过期待支付预约的应用服务能力。
- 超时取消必须通过状态条件更新预约，并且容量只释放一次。
- 手动取消和超时取消并发时不得重复释放容量。

### 2.3 预约核销与履约状态

- 核销模块识别 `RESERVATION` 业务类型后通知预约模块。
- 预约模块以 `CONFIRMED -> FULFILLED` 条件更新推进履约状态。
- `PENDING_PAYMENT`、`CANCELED`、`FULFILLED` 预约不得再次履约。
- 核销凭证状态、核销记录和预约状态必须保持一致。

### 2.4 顾客预约查询与取消

- 增加顾客预约列表接口，支持按联系电话和状态查询。
- 增加顾客预约详情接口。
- 增加顾客取消允许范围内预约的接口。
- 查询和取消必须使用服务端可信租户门店上下文，并校验顾客身份边界。

### 2.5 测试与验收

- 覆盖免费预约、付费预约、重复回调、错误金额和非法状态回调。
- 覆盖容量并发占用、释放、超时取消与手动取消并发边界。
- 覆盖预约核销重复提交和跨租户核销。
- 至少使用两个租户验证预约查询、取消、核销和容量不会串租户。

## 3. 当前实施顺序

首个功能单元从“预约支付与确认”开始，先完成后端持久化能力、回调接口、数据库迁移和最小自动化测试。真实微信证书、密钥和平台验签细节归第四阶段实现，第三阶段继续使用当前微信支付适配接口和本地替身。
