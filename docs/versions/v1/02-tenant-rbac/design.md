# 02 多租户、认证与 RBAC 阶段设计

## 1. 阶段目标

本阶段建立摊点通 V1 后续业务共享的安全底座，确保平台权限域与租户权限域隔离，业务请求中的租户、门店、用户和权限都来自服务端可信上下文。

## 2. 范围

- 建立平台用户、租户、门店、后台用户、角色、权限、用户角色、角色权限和操作日志基础表。
- 实现平台权限域与租户权限域枚举和用户上下文。
- 实现租户上下文与门店上下文的线程隔离。
- 实现密码哈希与校验，禁止明文密码持久化。
- 实现访问令牌签发与解析，令牌只标识用户和权限域，不信任客户端提交租户。
- 实现 RBAC 权限判定，接口鉴权以后端权限为准。
- 初始化 V1 默认权限标识和租户默认角色。
- 补充至少两个租户的隔离和权限判定自动化测试。

## 3. 不做事项

- 不实现商户自主入驻。
- 不支持动态数据权限。
- 不支持一个账号同时跨平台和租户权限域登录。
- 不支持租户管理员创建权限标识。
- 不实现前端登录页；统一后台页面在 03 阶段实现。

## 4. 安全模型

平台用户不绑定 `tenant_id`，只能访问 `/api/platform/v1/**`。租户后台用户必须绑定 `tenant_id` 和默认 `store_id`，只能访问 `/api/admin/v1/**`。C 端顾客会话在后续小程序阶段接入，本阶段只定义上下文能力。

访问令牌包含：

- 用户 ID。
- 权限域：`PLATFORM` 或 `TENANT`。
- 令牌版本。
- 签发时间和过期时间。

服务端解析令牌后，从数据库加载用户、租户、门店和权限。租户上下文只由服务端加载结果建立，禁止信任请求体、查询参数或前端传入的 `tenant_id`。

## 5. RBAC 权限

权限分为：

- `MENU`：菜单展示。
- `ACTION`：页面按钮和显式操作。
- `API`：后端接口安全边界。

API 鉴权必须校验 `API` 权限。菜单和按钮权限只改善体验，不能替代接口鉴权。

## 6. 租户隔离

租户业务表必须包含 `tenant_id`，门店相关表同时包含 `store_id`。本阶段先建立上下文和安全表索引；业务表隔离在后续业务阶段结合 MyBatis-Plus 租户拦截器和应用服务二次校验实现。

## 7. 验收标准

- 密码使用安全哈希，不记录明文。
- 平台和租户权限域不能混用。
- 租户上下文来自服务端用户归属。
- 至少两个租户的用户权限判定互不影响。
- 未授权访问返回统一错误结构。
- 阶段测试和静态检查通过。
