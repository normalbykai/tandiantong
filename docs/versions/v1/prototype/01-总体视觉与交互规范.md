# 摊点通 V1 总体视觉与交互规范

## 1. 设计目标

- C 端营造温暖、可信、具有线下烟火气的消费体验。
- B 端提供清晰、专业、高效率的管理体验。
- 平台与商户共用后台设计系统，但通过身份标识和菜单范围区分权限域。
- 所有界面文案必须使用中文，避免英文占位词和无意义中英文混排。

## 2. 基准画板

| 场景 | 基准尺寸 | 说明 |
| --- | --- | --- |
| C 端微信小程序 | `375 × 812` | 适配常见微信小程序屏幕 |
| B 端桌面后台 | `1440 × 900` | 左侧菜单、顶部栏、主内容区 |
| B 端移动后台 | `390 × 844` | 核销、订单查询和待办优先 |

移动端最小点击区域为 `44 × 44` 像素。桌面表格在移动端转换为卡片或折叠分区。

## 3. 色彩

### 3.1 C 端基础色

| 用途 | 建议色值 | 使用方式 |
| --- | --- | --- |
| 品牌暖橙 | `#F97316` | 主按钮、价格、重点入口 |
| 深暖橙 | `#C2410C` | 按下状态、强调文字 |
| 米白背景 | `#FFF8F0` | 页面背景 |
| 卡片白 | `#FFFFFF` | 商品、服务和凭证卡片 |
| 主文字 | `#2B2118` | 标题和正文 |
| 次文字 | `#7C6F64` | 辅助说明 |

商户主题色只替换主按钮、选中标签和少量装饰区域。系统必须检查主题色与白色文字的对比度，不合格时自动使用深色文字或推荐安全颜色。

### 3.2 B 端基础色

| 用途 | 建议色值 |
| --- | --- |
| 主色蓝绿 | `#0F766E` |
| 主色悬停 | `#115E59` |
| 页面背景 | `#F5F7FA` |
| 卡片与表格 | `#FFFFFF` |
| 主文字 | `#1F2937` |
| 次文字 | `#6B7280` |
| 边框 | `#E5E7EB` |

### 3.3 状态色

| 状态 | 色值 | 示例 |
| --- | --- | --- |
| 成功 | `#16A34A` | 已支付、已启用、已核销 |
| 警告 | `#D97706` | 待支付、库存预警、待复核 |
| 错误 | `#DC2626` | 支付失败、停用、审核驳回 |
| 信息 | `#2563EB` | 待处理、说明、处理中 |
| 中性 | `#6B7280` | 已取消、已失效、草稿 |

颜色不能成为唯一状态区分方式，必须同时显示中文文字或图标。

## 4. 字体与间距

- 优先使用系统中文字体：苹方、微软雅黑、思源黑体及系统无衬线字体。
- C 端页面标题 `20`，卡片标题 `16`，正文 `14`，辅助文字 `12`，取餐号可使用 `40` 至 `48`。
- B 端页面标题 `22`，区块标题 `16`，正文与表格 `14`，辅助文字 `12`。
- 间距采用 `4` 的倍数，常用为 `8、12、16、24、32`。
- C 端卡片圆角建议 `12`，B 端卡片圆角建议 `8`。

## 5. 核心组件

### 5.1 通用组件

- 主按钮、次按钮、文字按钮、危险按钮。
- 输入框、手机号输入、金额输入、日期时间选择。
- 状态标签、步骤条、分页签、空状态、错误提示、加载骨架。
- 二次确认弹窗、操作成功提示、权限不足提示。

### 5.2 C 端组件

- 商户头部、轮播图、商品卡片、服务卡片、分类标签。
- 规格与加料选择面板、购物车栏、价格明细。
- 日期栏、月历、时段按钮、剩余名额提示。
- 取餐凭证、预约凭证、核销二维码、营业状态条。

### 5.3 B 端组件

- 左侧菜单、顶部身份栏、数据指标卡、待办卡片。
- 筛选区、桌面表格、移动卡片、详情摘要、详情页签。
- 分步表单、审核清单、库存调整弹窗、核销确认面板。
- 权限树、角色表格、操作日志时间线、导出任务状态。

## 6. 状态与反馈

每个页面必须定义以下状态：

- 加载中：使用骨架或局部加载，禁止整页长时间空白。
- 空数据：说明当前没有数据，并提供符合权限的下一步操作。
- 请求失败：说明失败原因和重试入口，不暴露内部异常。
- 无权限：显示“暂无访问权限”，提供返回入口。
- 禁用：按钮保留但不可点击时必须说明原因。
- 提交中：按钮显示处理中并防止重复提交。
- 成功：给出明确结果和下一步入口。

## 7. 表单规范

- 必填项使用中文“必填”提示，不只使用红色星号。
- 校验信息紧邻字段展示，例如“请输入正确的手机号”。
- 长表单支持保存草稿；步骤切换不得丢失已填写内容。
- 危险操作必须二次确认，并说明影响范围。
- 支付配置、退款、停用租户、库存调整和核销必须记录操作原因或审计信息。

## 8. 中文文案规范

- 页面标题采用“动作或对象 + 场景”，例如“新建商户”“订单详情”。
- 按钮使用明确动作，例如“保存草稿”“提交复核”“确认核销”。
- 禁止使用“确定”处理含义不清的复杂操作。
- 错误提示说明发生了什么以及用户能做什么。
- 示例商户使用“春风小铺”，商品使用“桂花拿铁”“手作三明治”，服务使用“咖啡体验课”等自然中文数据。
