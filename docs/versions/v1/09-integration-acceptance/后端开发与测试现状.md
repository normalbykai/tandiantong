# V1 后端开发与测试现状

## 1. 文档目的

本文记录截至 2026-07-12 的 V1 后端实际完成情况，用于后续人工测试、继续开发和验收判断。本文只描述已经存在于 `codex/v1-development` 分支中的实现，不把静态前端原型或未提交草稿视为后端已完成能力。

## 2. 总体结论

- 后端工程基础、平台与租户认证、商户开通、商品库存、商品订单支付退款、免费预约、商品核销、经营数据和 Excel 导出已经形成主要业务闭环。
- 当前不能认定 V1 后端全部完成。加料成交快照、订单超时释放、顾客微信登录、付费预约、预约核销、数据库 RBAC 权限码校验、退款失败重试和完整审计仍未实现。
- 已执行的 Maven 全量测试通过；部分交易流程已连接本地 MySQL 进行真实接口验证，但双租户、并发和异常重试场景仍缺完整自动化集成测试。
- 微信支付使用安全的本地模拟适配器。没有真实微信商户凭据，因此不得视为真实微信支付接入成功。

## 3. 已实现且已验证

### 3.1 工程基础

- Java 21、Spring Boot 3、Maven 多模块模块化单体工程已建立。
- 统一 API 响应、中文业务错误、追踪号、全局异常处理和基础安全配置已建立。
- MySQL 8、Redis 和 Docker Compose 基础开发环境已启动验证。
- Flyway 迁移 `V1_0_0` 至 `V1_8_0` 已在本地 MySQL 成功执行。
- 最近一次执行 `mvn -q test` 成功，启动模块接口契约测试和各业务模块单元测试均通过。

### 3.2 多租户、认证与商户开通

- 平台管理员 JWT 登录已通过真实接口和数据库验证。
- 平台创建商户、默认门店、管理员邀请、支付配置占位和小程序入口码已通过真实写库验证。
- 管理员邀请码激活和租户管理员登录已通过真实接口验证。
- 商户启用接口和平台商户列表接口已实现并通过接口契约测试。
- JWT 只保存用户 ID、权限域和 token version；租户、门店和用户状态由数据库重新查询。
- 平台和租户 API 分别要求 `ROLE_PLATFORM` 和 `ROLE_TENANT`。

### 3.3 商品、SKU 与库存

- 商品、分类、SKU、初始库存和库存流水创建接口已实现。
- 金额统一使用整数分，商品成交价格从服务端 SKU 读取。
- 后台商品列表和库存流水查询接口已实现，并使用服务端租户、门店上下文限定范围。
- 小程序商品列表通过不可伪造的 `scene_key` 反查租户和门店。
- 库存锁定使用 `available_stock >= quantity` 条件更新，防止超卖。
- 商品、SKU 和初始库存流水已通过真实接口写库验证。

### 3.4 商品订单、支付与退款

- 商品订单创建、服务端计价、订单项目快照、库存锁定和创建幂等已实现。
- 本地微信预支付适配、支付回调签名验证、回调幂等和支付记录已实现。
- 支付成功后确认库存扣减，并生成商品核销凭证。
- 核销前整单退款、退款幂等、库存回补和凭证作废已实现。
- 真实数据库验证确认：重复下单只锁一次库存；重复支付回调只产生一条支付记录；重复退款只产生一条退款记录。

### 3.5 免费预约

- 服务项目、预约时段和容量配置接口已实现。
- 免费预约创建、幂等、容量原子占用和取消释放容量已实现。
- 容量为 1 时，第二次预约返回业务错误；取消后容量恢复，已通过真实接口和数据库验证。

### 3.6 商品核销

- 商品取餐号和不可猜测核销 token 已实现。
- 数据库只保存 token 的 SHA-256 摘要，支付首次成功响应只返回一次 token 原文。
- 商品核销使用带当前状态条件的原子更新，重复核销最多成功一次。
- 已退款订单的核销凭证会被作废。

### 3.7 经营数据与导出

- 订单、实收、退款、待核销、预约和商品排行聚合接口已实现。
- Excel 导出已实现，文件为有效 XLSX，文件头验证为 `PK`。
- 导出联系人会脱敏，导出任务和中文审计说明会写入数据库。
- 经营数据查询和 Excel 导出已通过真实接口与数据库验证。

## 4. 已实现但验证不足

- 商品核销已有单元测试和原子实现，但缺少连接 MySQL 的完整端到端重复核销、并发核销测试。
- 租户隔离已落实在主要 SQL、JWT 上下文和 `scene_key` 反查中，但缺少覆盖查询、更新、导出和缓存的双租户自动化集成测试。
- 库存锁定和预约容量使用数据库条件更新，但缺少多线程并发集成测试。
- 支付与退款正常和重复请求已验证，但缺少外部超时、退款失败、重试上限和人工处理状态测试。
- Excel 导出内容和文件格式已验证，但缺少大数据量、日期边界和跨租户导出测试。
- Redis 基础连接可用，但尚未形成实际业务缓存、锁或幂等键的双租户隔离验证。
- 平台商户列表和后台商品库存查询已通过接口契约测试，尚未纳入完整人工后台操作测试。

## 6. 建议人工测试清单

1. 使用平台管理员登录，创建两个不同商户，分别激活管理员并启用商户。
2. 分别以两个租户登录，创建同名商品和 SKU，确认查询结果、库存流水和导出不会串租户。
3. 使用租户 A 的 `scene_key` 下单，尝试提交租户 B 的 SKU，确认后端拒绝。
4. 对同一幂等键重复下单，确认订单号一致且库存只锁定一次。
5. 对同一支付回调重复请求，确认支付记录、库存确认扣减和凭证只生成一次。
6. 使用正确 token 核销两次，确认第一次成功、第二次返回一致的已核销结果或业务错误。
7. 对待核销订单重复退款，确认只生成一条退款记录并只回补一次库存。
8. 创建容量为 1 的预约时段并连续预约两次，确认第二次失败；取消后再次预约应成功。
9. 导出两个租户的经营数据，核对文件内容、审计记录和联系人脱敏。
10. 停用租户后尝试登录和发起业务写操作，确认新交易被阻止且历史数据仍可保留查询。
