# 08 经营数据设计

## 1. 阶段目标

在订单、预约和核销事实数据稳定后，为商户权限域提供当前租户内的基础经营数据。V1 只覆盖交易概览、商品分析、预约分析和 Excel 导出审计，不建设数据仓库，不提供平台跨商户查看经营数据入口。

## 2. 数据边界

- 报表查询必须显式使用服务端可信的 `tenantId` 和 `storeId`。
- 交易指标按营业日汇总订单数、已支付订单数、实收金额、退款金额、净收入和待核销数量。
- 商品指标按商品、SKU 和加料项聚合销量排行，金额使用整数“分”。
- 预约指标按预约数、取消数、履约数和时段使用率聚合。
- 导出任务记录租户、门店、导出类型、日期范围、脱敏操作人、文件名和审计说明。

## 3. 模块设计

`tandiantong-analytics` 提供独立应用服务 `AnalyticsService`。当前阶段沿用前序阶段的内存事实模型与单元测试，数据库迁移同步建立后续持久化所需事实表和导出审计表。业务模块不得直接跨库读取其他模块实现，后续接入真实持久化时通过应用服务写入报表事实。

## 4. 安全与合规

- 平台权限域不展示商户经营金额和图表。
- 导出权限独立于查看权限，缺少导出权限时拒绝创建任务。
- 导出联系人和敏感字段默认脱敏，审计记录使用中文说明。
- 当前条件没有数据时拒绝生成空导出任务，避免形成误导性文件。

## 5. 不做事项

- 不实现定时报表、邮件发送和 BI 看板。
- 不引入搜索引擎、消息队列、数据仓库或异步任务基础设施。
- 不实现多门店合并报表和平台查看商户经营数据。
