# 摊点通 V1 分层实施路线图

## 1. 文档目标

本文档用于在正式开发前确认摊点通 V1 的实施方向。V1 按“先底座、再安全、再交易闭环、最后运营与验收”的顺序推进，避免在系统尚未稳定前提前堆叠未来能力。

本路线图只覆盖 `docs/versions/v1/README.md` 中定义的第一阶段范围，不扩展配送、多门店、套餐、人员排班、平台分账、自由装修、消息队列、微服务和 Kubernetes 等未来能力。

## 2. 分层原则

- 基础层先行：工程结构、配置、数据库迁移、统一响应、异常和日志必须先完成。
- 安全层先行于业务层：多租户上下文、认证、RBAC 和审计能力必须早于商品、订单和预约业务。
- 交易闭环优先于报表体验：商品、库存、支付、退款、核销和预约状态必须先保证正确性、幂等性和并发安全，再建设经营数据。
- 每层必须可独立验证：完成一层后运行对应测试并形成阶段验收记录，禁止以未验证状态进入下一层。
- 文档与实现同步：每个阶段开始前补齐 `design.md`、必要的 `prototype.md`、`implementation-plan.md` 和 `acceptance.md`。

## 3. 实施层级总览

| 实施层级 | 对应阶段 | 主要目标 | 进入下一层条件 |
| --- | --- | --- | --- |
| L0 品牌与项目基线 | 00 总览与原型 | 确认产品边界、视觉基线、Logo 资产和开发规则 | 文档入口清楚，Logo 可用于后续原型和前端 |
| L1 工程基础层 | 01 工程基础 | 建立后端模块化单体、数据库迁移、统一基础能力和测试骨架 | 应用可启动，最小测试与迁移可执行 |
| L2 安全与租户层 | 02 多租户与 RBAC | 建立可信租户上下文、认证、平台与租户权限域、角色权限 | 两租户隔离和 API 鉴权通过测试 |
| L3 平台开通层 | 03 统一后台与商户开通 | 支持平台创建商户、初始化门店、邀请管理员、配置品牌和小程序码 | 商户从创建到启用流程可走通 |
| L4 商品库存层 | 04 商品与库存 | 支持分类、商品、SKU、加料、上下架、库存和库存流水 | 商品维护和库存变更具备租户隔离与流水追踪 |
| L5 订单支付层 | 05 订单与支付 | 支持购物车、订单快照、库存锁定、微信支付、退款和幂等 | 商品购买到支付成功、取消、退款流程稳定 |
| L6 预约履约层 | 06 服务预约、07 取餐号与核销 | 支持服务、时段容量、预约凭证、商品和预约核销 | 重复预约、重复回调和重复核销不会重复改变状态 |
| L7 运营数据层 | 08 经营数据 | 支持交易、商品、预约基础报表和 Excel 导出 | 报表查询有索引、权限和导出审计 |
| L8 联调验收层 | 09 联调与验收 | 完成全链路测试、安全检查、部署验证和 V1 验收 | 用户确认 V1 可以进入正式开发收尾或发布准备 |

## 4. L0 品牌与项目基线

### 4.1 目标

确认开发前的共同基线，确保后续阶段围绕同一套产品范围、架构边界、视觉规范和品牌资产推进。

### 4.2 交付物

- V1 架构设计：`docs/versions/v1/00-overview/architecture.md`。
- V1 产品原型：`docs/versions/v1/prototype/`。
- 系统主 Logo：`docs/versions/v1/assets/brand/tandiantong-logo-full.png`。
- 系统图标 Logo：`docs/versions/v1/assets/brand/tandiantong-logo-icon.png`。

### 4.3 Logo 使用规则

V1 选择彩色横版“图标 + 中文名称 + 英文名称”作为系统主 Logo。主 Logo 用于后台登录页、系统顶部栏、文档封面和正式品牌展示；单独图标用于 favicon、小程序图标草案、移动端紧凑导航和加载状态。

后续如需生成透明底、深色底、单色或小程序正式上传版本，应在 `docs/versions/v1/assets/brand/` 下新增派生文件，禁止直接覆盖原始 Logo 文件。

## 5. L1 工程基础层

### 5.1 目标

建立可持续开发的工程骨架，让所有后续业务模块共享统一启动方式、基础响应、异常处理、日志追踪、数据库迁移和测试规范。

### 5.2 范围

- 创建 Maven 多模块结构。
- 创建 Spring Boot 启动模块和环境配置骨架。
- 建立统一响应结构、分页模型、业务异常和错误码。
- 建立请求追踪号、中文业务日志和敏感信息保护规则。
- 接入 Flyway，创建基础迁移目录和空库验证流程。
- 建立单元测试、模块集成测试和静态检查命令。

### 5.3 不做事项

- 不实现具体业务表。
- 不提前创建未来模块空壳。
- 不接入消息队列、注册中心、配置中心或微服务框架。

### 5.4 验收重点

- 后端应用可以在本地配置下启动。
- Maven 模块不存在循环依赖。
- Flyway 可以执行空库初始化。
- 统一异常不会向客户端暴露堆栈、SQL 或敏感配置。

## 6. L2 安全与租户层

### 6.1 目标

建立所有业务能力依赖的安全底座，确保平台管理员和租户用户权限域分离，所有租户业务访问都来自服务端可信上下文。

### 6.2 范围

- 后台账号登录、访问令牌和刷新令牌。
- 平台用户与租户用户独立认证。
- 租户上下文、门店上下文和用户上下文。
- Spring Security 统一鉴权入口。
- RBAC 权限模型、默认角色、权限初始化和授权校验。
- 操作日志和关键权限变更审计。
- MyBatis-Plus 多租户隔离基础能力。

### 6.3 不做事项

- 不支持一个账号跨平台和租户权限域登录。
- 不支持动态数据权限。
- 不支持租户管理员创建权限标识。

### 6.4 验收重点

- 至少两个租户的查询和更新互不影响。
- 客户端提交的 `tenant_id` 和 `store_id` 不会被信任。
- 菜单和按钮隐藏不替代后端 API 鉴权。
- 租户停用后不能继续产生新交易或业务写操作。

## 7. L3 平台开通层

### 7.1 目标

让平台能够创建和启用商户，形成后续商品、订单和预约业务的租户入口。

### 7.2 范围

- 统一后台登录页的平台管理与商户管理入口。
- 平台商户列表、新建商户向导和商户详情。
- 租户、门店、首个租户管理员和默认角色初始化。
- 管理员邀请激活流程。
- 商户展示资料、Logo、头图、主题色和首页模板配置。
- 默认小程序码入口标识管理。
- 商户启用、停用和审核记录。

### 7.3 不做事项

- 不实现商户自主入驻。
- 不实现平台分账。
- 不实现渠道码和来源统计。

### 7.4 验收重点

- 商户从创建、邀请管理员、补齐资料到启用可以完整走通。
- 小程序码入口不直接暴露 `tenant_id`。
- 支付未配置时禁止上架付费商品和付费服务。
- 停用租户保留历史数据但阻止新业务写入。

## 8. L4 商品库存层

### 8.1 目标

建立商品售卖基础，让商户可以维护分类、商品、SKU、加料项和库存，并为订单创建提供稳定快照来源。

### 8.2 范围

- 商品分类、商品基础信息、图片、上下架状态。
- SKU 规格、价格、库存和编码。
- 加料分组、加料项和商品关联。
- 商品分步编辑和 C 端预览。
- 入库、出库、盘点和库存流水。
- 商品查询接口和小程序商品展示接口。

### 8.3 不做事项

- 不支持套餐商品。
- 不支持多门店库存。
- 不支持采购、供应商和调拨。

### 8.4 验收重点

- 金额统一使用整数“分”。
- 商品、SKU 和加料变化不会影响历史订单快照。
- 所有库存变化必须有流水。
- 商品唯一约束包含 `tenant_id`，必要时包含 `store_id`。

## 9. L5 订单支付层

### 9.1 目标

打通商品购买交易闭环，保证订单、库存、支付、退款和异常重试在并发和重复请求下保持一致。

### 9.2 范围

- 当前商户购物车。
- 创建订单、订单快照和订单状态机。
- 待支付订单库存锁定。
- 支付超时取消和库存释放。
- 微信支付下单、回调验签和支付记录。
- 核销前整单退款、退款记录和库存回补。
- 支付查询、退款查询和有限重试任务。

### 9.3 不做事项

- 不支持部分退款。
- 不支持核销后在线退款。
- 不依据前端跳转结果修改支付状态。

### 9.4 验收重点

- 重复创建、重复支付回调和重复退款请求返回一致业务结果。
- 支付回调通过平台订单号反查租户。
- 事务内不执行不可控的长时间网络调用。
- 订单、支付、退款和状态日志不物理删除。

## 10. L6 预约履约层

### 10.1 目标

完成服务预约和核销履约能力，让商品订单和服务预约都具备安全凭证、状态流转和并发防重。

### 10.2 范围

- 服务项目、免费或付费规则。
- 日期、时段、容量和暂停预约。
- 免费预约确认和付费预约支付后确认。
- 预约凭证和核销二维码。
- 商品取餐号、商品核销凭证和核销记录。
- B 端扫码核销、取餐号查询和待核销列表。
- 重复核销结果展示。

### 10.3 不做事项

- 不支持人员排班、房间和设备资源。
- 不支持取餐号作为安全凭证。
- 不支持已完成订单或已履约预约通过普通接口回退状态。

### 10.4 验收重点

- 预约容量通过条件更新、版本控制或等价原子方式占用。
- 商品和预约核销必须带当前状态条件执行原子更新。
- 并发或重复核销最多成功一次。
- 免费预约和付费预约凭证生成时机符合架构设计。

## 11. L7 运营数据层

### 11.1 目标

在交易事实数据稳定后提供基础经营分析，帮助商户查看订单、商品、预约和退款情况。

### 11.2 范围

- 商户工作台关键指标。
- 订单数、实收金额、退款金额和待核销数量。
- 商品、SKU 和加料项销量排行。
- 预约数、取消数、履约数和时段使用率。
- Excel 导出任务、下载和审计记录。

### 11.3 不做事项

- 不建设数据仓库。
- 不实现定时报表和邮件发送。
- 不提供平台查看商户经营数据入口。

### 11.4 验收重点

- 报表查询显式携带租户条件并使用必要索引。
- 导出权限独立控制，敏感字段默认脱敏。
- 大数据量导出使用后台任务并记录审计。

## 12. L8 联调验收层

### 12.1 目标

完成 V1 全链路质量验证，确认系统可以进入用户验收和发布准备。

### 12.2 范围

- C 端扫码进入商户、下单、支付、取餐凭证和核销。
- C 端免费预约、付费预约、预约凭证和核销。
- B 端平台创建商户、启用、停用和审计。
- B 端商品维护、库存调整、订单退款和经营数据。
- 租户隔离、RBAC、敏感信息、支付回调和并发边界测试。
- Docker、Nginx、MySQL、Redis 和对象存储部署验证。

### 12.3 验收重点

- 阶段范围内自动化测试通过。
- 关键业务链路完成联调记录。
- 已知限制和未完成项写入验收文档。
- 用户确认后再决定合并、创建 Pull Request 或保留分支。

## 13. 阶段文档创建顺序

每个阶段开始时只创建当前阶段需要的文档，禁止提前生成大量空文档。

```text
阶段开始
→ 编写 design.md
→ 如涉及页面交互，编写 prototype.md
→ 编写 implementation-plan.md
→ 按可独立验证的功能单元开发和提交
→ 编写 acceptance.md
→ 用户验收
→ 进入下一阶段
```

阶段内文档必须聚焦本阶段边界，不得把后续阶段的详细实现持续堆叠进同一份文档。

## 14. 第一批开发建议

正式进入开发时，建议先启动 `01-foundation/` 阶段，最小功能单元如下：

1. 创建 Maven 多模块骨架和 Spring Boot 启动模块。
2. 建立统一响应、错误码、业务异常和请求追踪号。
3. 接入 Flyway、MySQL、Redis 基础配置和测试容器或本地测试配置。
4. 建立最小测试命令和模块依赖约束检查。
5. 补齐 `01-foundation/design.md`、`implementation-plan.md` 和 `acceptance.md`。

该阶段完成并验证后，再进入 `02-tenant-rbac/`，避免在安全底座未成形前实现交易业务。
