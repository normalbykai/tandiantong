# 摊点通项目级规则设计

## 1. 目标

在项目根目录创建单一 `AGENTS.md`，作为后续人工开发和智能编码代理共同遵循的项目级强制规范。规则应简洁、可执行，并与已确认的第一阶段架构设计保持一致。

## 2. 作用范围

根目录 `AGENTS.md` 适用于整个仓库，包括 Java 后端、管理后台、小程序、数据库脚本、测试、配置和文档。若未来某个子目录确有特殊约束，可增加更具体的子目录规则，但不得削弱根规则中的租户安全、支付安全和敏感信息保护要求。

## 3. 规则结构

正式规则文件包含以下内容：

1. 项目目标与第一阶段范围，防止无需求扩展。
2. 中文注释规范，明确必须注释和无需注释的场景。
3. Maven 模块职责、依赖方向和跨模块调用约束。
4. Java、TypeScript、接口、异常和日志编码规范。
5. 多租户、RBAC、金额、库存、支付、退款和核销安全规则。
6. MySQL、Redis、Flyway 和数据变更规范。
7. 测试、验证和完成标准。
8. 配置、密钥、日志脱敏和 Git 变更安全规范。

## 4. Git 提交信息规范

- 后续人工创建的 Git 提交信息必须使用简体中文描述，不重写已有提交历史。
- 允许使用 `feat:`、`fix:`、`docs:`、`refactor:`、`test:`、`build:`、`chore:` 等英文类型前缀，但冒号后的主题必须使用中文。
- 功能、修复、重构、文档、测试、构建和配置等人工提交不得使用纯英文主题。
- Git 自动生成的合并提交，以及拉取或同步代码产生的提交，可以保留默认英文信息。
- 技术名词、模块名、类名和缺陷编号可以保留英文，但提交信息的主要描述必须为中文。
- 禁止只写类型前缀、编号或无意义短语来规避中文描述要求。
- 合规示例：`feat: 增加商品库存锁定功能`、`fix: 修复支付回调重复扣减库存问题`。
- 不合规示例：`feat: add inventory lock`、`fix: payment bug`。

### 4.1 开发与提交粒度

- 开发任务应当先按业务阶段拆分，再将阶段内工作拆为可独立验证的功能单元。
- 每完成一个功能单元并通过相关验证后应当及时提交，禁止等整个项目或整个阶段全部完成后只创建一个巨大提交。
- 禁止按文件、方法或无业务意义的小改动机械拆分提交，避免产生大量无法独立理解和验证的碎片提交。
- 一个功能单元提交应当聚焦单一目的，并尽量同时包含必要的数据库迁移、实现、测试和相关文档。
- 每次功能提交必须至少能够正常编译，并通过与改动直接相关的测试和静态检查；禁止提交已知不可运行的半成品。
- 阶段完成后必须执行阶段范围内的整体测试和联调，由用户确认后再进入下一阶段。
- 默认在 `codex/` 前缀的功能分支中开发，阶段完成后由用户决定合并、创建 Pull Request 或保留分支。

## 5. 中文注释要求

- 必要的代码注释、JavaDoc、复杂 SQL 注释和配置说明必须使用简体中文。
- 禁止新增英文解释性注释，也不允许中英文混写同一解释性注释。
- 类名、方法名、变量名、数据库字段、接口路径和标准技术术语仍使用规范英文命名。
- 注释用于解释业务规则、设计原因、状态流转、并发控制、安全边界和不直观实现。
- 不为简单赋值、普通分支、明显方法或重复代码含义编写无价值注释。
- 修改代码后必须同步更新已经失效的注释，禁止保留与实现不一致的描述。

### 5.1 中文产品文档要求

- 原型、需求、产品设计、业务流程、验收说明和人工智能原型提示词的正文及界面文案必须使用简体中文。
- 页面名称、按钮、字段、状态、提示、线框图标注和示例数据必须使用自然中文。
- `README.md`、`AGENTS.md` 等约定文件名可以保留英文，但正文必须使用中文。
- API 路径、代码标识、数据库字段、类名和标准协议可以保留英文；技术术语首次出现时应提供中文含义。
- 禁止纯英文页面、英文占位文案和无意义的中英文混排。

## 6. 强制级别

使用“必须”“禁止”“应当”“可以”表达规则强度：

- “必须”和“禁止”属于不可违反的项目约束。
- “应当”是默认实践，只有明确理由时才可偏离。
- “可以”表示可选做法，不作为交付要求。

规则强调实际安全边界和可维护性，不强制无意义封装、过度抽象、过量注释或与第一阶段无关的基础设施。

## 7. 一致性要求

`AGENTS.md` 必须引用文档总索引 `docs/README.md`、当前版本索引 `docs/versions/v1/README.md` 和 V1 架构文档 `docs/versions/v1/00-overview/architecture.md`。当规则、版本范围、阶段文档或架构发生冲突时，应先停止实现并更新文档形成明确结论，不得自行选择更宽松的解释。

## 8. 验收标准

- 规则覆盖整个仓库且位置正确。
- 中文注释要求没有歧义。
- 多租户隔离和交易幂等被列为强制规则。
- Git 人工提交的中文主题要求和允许的例外场景没有歧义。
- 提交粒度同时避免巨大提交和无意义碎片提交，并明确阶段验收要求。
- 不包含尚未确认的产品功能或技术组件。
- 不存在 `TODO`、`TBD` 或未决占位内容。
- 内容足够具体，可用于开发检查和代码评审。
