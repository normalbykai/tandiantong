# 摊点通第一阶段架构设计

## 1. 项目概述

摊点通（英文名 `tandiantong`）是一套面向线下零售与到店服务商户的多租户系统。第一阶段包含顾客端微信小程序、商户响应式 Web 管理后台和平台管理能力。

顾客通过商户专属小程序码进入指定商户，可购买商品或预约服务。商品支付成功后生成取餐号和核销凭证，预约确认后生成预约核销凭证。商户在管理后台维护商品、服务和基础运营数据，并通过手机浏览器或桌面浏览器执行核销。

第一阶段按单租户单门店交付，核心数据预留 `store_id`，但不实现多门店切换、汇总或数据权限。

## 2. 第一阶段范围

### 2.1 范围内

- 平台管理员手动创建、停用租户，并初始化门店和首个租户管理员。
- 顾客扫描商户专属小程序码进入对应租户，不提供商户广场、搜索和附近门店。
- 商品分类、商品、SKU、加料项、价格、库存和上下架管理。
- 微信支付、到店自提、取餐号、核销二维码、整单退款。
- 免费预约和付费预约，按服务项目、日期、时段和容量管理。
- 响应式商户 Web 后台，支持桌面运营和手机端核销。
- 默认角色、租户自定义角色，以及菜单、操作和 API 权限。
- 基础订单、商品、加料项和预约运营数据。

### 2.2 暂不实现

- 商户自主入驻和资质审核。
- 多门店运营、跨门店库存和跨门店报表。
- 堂食桌号、外卖配送、地址、运费和骑手履约。
- 套餐商品、员工排班、房间或设备资源预约。
- 部分退款、复杂售后和核销后在线退款。
- 按门店、商品分类、订单归属等动态数据权限。
- 消息队列、搜索引擎、微服务、注册中心、配置中心和 Kubernetes。

## 3. 总体架构

采用基于 Maven 多模块的模块化单体。所有模块编译为一个 Spring Boot 应用统一部署，模块之间通过应用服务或明确接口协作，禁止跨模块直接访问对方的数据访问层。

```text
tandiantong/
├─ tandiantong-bootstrap        启动、配置与统一装配
├─ tandiantong-common           通用响应、异常、分页、日志与基础工具
├─ tandiantong-security         登录、用户、租户上下文与 RBAC
├─ tandiantong-catalog          分类、商品、SKU、加料、上下架与库存
├─ tandiantong-order            购物车、订单、支付与退款
├─ tandiantong-reservation      服务项目、时段、容量与预约
├─ tandiantong-verification     取餐号、核销凭证与核销记录
├─ tandiantong-analytics        基础经营指标与报表查询
├─ tandiantong-integration      微信登录、微信支付与对象存储适配
├─ tandiantong-admin-api        商户后台接口
└─ tandiantong-mini-api         顾客小程序接口
```

依赖方向如下：

```text
admin-api ─┐
mini-api  ─┼─> 业务模块 ─> common
bootstrap ─┘       │
                   └─> integration
```

API 模块只负责协议适配、参数校验和响应转换。业务规则放在相应业务模块，第三方 SDK 和外部服务调用集中在 `integration` 模块。

## 4. 技术选型

| 层次 | 技术 |
| --- | --- |
| C 端 | uni-app、TypeScript；第一阶段仅验收微信小程序 |
| B 端 | Vue 3、TypeScript、Vite、Element Plus |
| 后端 | Java 21、Spring Boot 3、Spring Security、Maven 多模块 |
| 数据访问 | MyBatis-Plus，结合明确的数据访问规范 |
| 数据库 | MySQL 8、Flyway |
| 缓存与并发辅助 | Redis |
| 文件 | 兼容 S3 协议的对象存储 |
| 部署 | Docker、Nginx、单个 Spring Boot 应用 |

当前目标规模为 10 至 100 个租户，日订单总量低于 1 万。架构在数据模型和模块边界上为中等规模增长留出空间，但不提前实施分布式基础设施。

## 5. 多租户隔离

第一阶段采用共享数据库、共享数据表和 `tenant_id` 行级隔离。门店相关业务表同时包含 `store_id`。

租户上下文必须来自可信来源：

- C 端由商户小程序码 `scene` 参数解析租户，服务端验证后写入用户会话。
- B 端从已登录后台用户的租户归属建立上下文，不信任客户端提交的 `tenant_id`。
- 支付回调通过平台订单号反查租户，不依赖登录会话或回调请求中的租户参数。
- 平台管理员位于独立平台权限域，不伪装成普通租户用户。

系统采用双层防护：MyBatis-Plus 多租户拦截器自动追加租户条件；订单、支付、退款、核销等敏感应用服务再次校验资源归属。定时任务必须显式携带租户上下文，缓存键必须包含 `tenant_id`。

所有租户业务唯一索引包含租户维度，例如 `(tenant_id, order_no)`、`(tenant_id, pickup_no, business_date)` 和 `(tenant_id, sku_code)`。

租户停用后，C 端和 B 端均不得创建新交易或继续业务操作。历史数据仍保留并仅供具备平台审计授权的能力访问；第一阶段不提供平台查看商户经营数据的入口。

## 6. RBAC 权限模型

租户后台采用“用户、角色、权限”模型：

```text
后台用户 ──< 用户角色 >── 角色 ──< 角色权限 >── 权限
                  │
               tenant_id
```

权限分为菜单权限、操作权限和 API 权限。菜单和按钮只改善界面体验，后端 API 权限是最终安全边界。权限由平台预定义；租户管理员可以创建角色并组合其本人拥有的权限，但不能创建权限标识或越权授权。

默认租户角色为：

- 租户管理员：租户内全部权限，可管理员工和角色。
- 运营人员：商品、预约配置和经营数据权限。
- 核销人员：待履约查询和核销权限。

角色、权限关系和关键管理操作均记录租户、操作人、变更内容和时间。第一阶段不支持动态数据权限表达式。

## 7. 商品与库存

商品域支持分类、商品、SKU 和加料项。一个商品可配置多个规格维度并生成 SKU；加料项可分组并与商品关联，支持额外加价。第一阶段不支持套餐。

下单时由服务端校验商品、SKU、加料项、上下架状态和库存，并统一计算金额。订单明细保存商品、SKU、加料名称和成交价格快照，商品后续修改不会影响历史订单。金额统一使用整数“分”，禁止使用浮点数。

库存变化必须写入库存流水。创建待支付订单时短时锁定库存；支付成功后确认扣减，订单取消或支付超时后释放。核销前整单退款成功后自动回补可售库存，并记录退款原因和库存流水。

## 8. 商品订单、支付与核销

商品订单状态机为：

```text
待支付 → 待核销 → 已完成
   └→ 已取消     └→ 退款中 → 已退款
```

顾客创建订单后，系统保存订单快照并锁定库存。微信支付回调必须验签并幂等处理；支付成功后订单进入待核销状态，并按“租户 + 营业日”生成唯一取餐号。

取餐号用于顾客和商户识别，不作为安全凭证。系统同时生成不可猜测的核销令牌并形成二维码。核销通过带状态条件的数据库更新完成，确保并发或重复请求最多成功一次。

待支付订单超时后自动取消并释放库存。第一阶段仅支持核销前整单退款；已完成订单不能在系统内发起在线退款。微信重复回调、重复退款查询和重复核销均返回稳定业务结果，不重复改变状态或库存。

## 9. 服务预约

预约按服务项目、日期、时段和容量管理，不区分服务人员、房间或设备。服务项目可以配置为免费预约或微信支付预约。已确认预约在时段开始前允许取消；免费预约取消后直接释放容量，付费预约在整单退款成功后释放容量。时段开始后只能由商户核销或进行线下异常处理，第一阶段不提供在线取消和退款。

预约状态机为：

```text
待支付 → 已预约 → 已履约
   └→ 已取消  └→ 已取消

免费预约直接进入“已预约”
```

创建预约时通过版本号或条件更新占用时段容量，禁止超额预约。免费预约成功后直接生成预约凭证；付费预约在支付成功后生成。取消或支付超时释放时段容量。核销完成后预约进入已履约状态，不能继续在线退款。

## 10. 核心数据分组

```text
租户权限：
tenant、store、admin_user、role、permission、
user_role、role_permission、operation_log

商品：
category、product、product_sku、
addon_group、addon_option、product_addon_relation、inventory_record

订单支付：
customer、sales_order、sales_order_item、sales_order_item_addon、
payment_record、refund_record、order_status_log

预约：
service_item、service_schedule、service_slot、
reservation、reservation_status_log

核销：
verification_credential、verification_record
```

订单、支付、退款、预约和核销等交易记录不得物理删除。状态变化保留流水或状态日志，支持异常追踪和审计。

## 11. API 与认证

接口按调用方分区并使用版本前缀：

```text
/api/mini/v1/**      顾客小程序
/api/admin/v1/**     商户后台
/api/platform/v1/**  平台后台
/api/callback/**     微信支付回调
```

C 端通过微信登录 `code` 换取系统用户会话。B 端使用账号密码登录，签发短期访问令牌和可撤销刷新令牌。令牌只标识用户，会话中的租户和权限由服务端校验。支付回调独立验签，不经过普通登录认证。

接口使用统一业务错误码和请求追踪号。客户端只获得适合展示的信息，服务端日志保留可定位问题的异常上下文，但不得记录密码、完整令牌、完整支付敏感参数或其他敏感信息。

## 12. 运营数据

首期提供以下租户级指标：

- 指定日期范围的订单数、实收金额和退款金额。
- 待核销、已核销和已退款订单数量。
- 商品、SKU 和加料项的销量及销售额排行。
- 预约数、取消数、履约数和时段使用率。

当前规模下直接使用经过索引优化的 MySQL 聚合查询。第一阶段不建设数据仓库、搜索引擎或独立报表服务。

## 13. 异步任务与异常处理

支付查询、退款重试等少量异步任务使用数据库任务表和定时扫描。任务记录重试次数、下次执行时间和最终状态，并通过业务幂等键防止重复执行。第一阶段不引入消息队列。

外部服务调用必须设置超时并区分可重试和不可重试错误。支付回调处理失败时由微信重试，系统同时保留异常记录供定时查询和后台排查。任何外部调用失败均不得使本地业务状态进入无法解释的中间状态。

## 14. 部署与运维

```text
Nginx
├─ C 端与 B 端静态资源
└─ Spring Boot 模块化单体
   ├─ MySQL 8
   ├─ Redis
   └─ 对象存储
```

应用和前端通过 Docker 部署。系统提供健康检查、结构化日志、请求追踪号和支付、退款、任务失败等基础告警。MySQL 定期备份，并在上线前验证恢复流程。数据库结构变更由 Flyway 管理。

## 15. 测试策略

- 单元测试覆盖价格计算、SKU 与加料组合、状态机和 RBAC 判断。
- 模块集成测试覆盖库存锁定与释放、预约容量、支付、退款和核销。
- 租户隔离测试使用至少两个租户验证查询、更新、缓存和导出不会串租户。
- 并发测试覆盖最后一件库存、最后一个预约名额和重复核销。
- 微信适配测试覆盖登录、支付、回调验签、重复回调和退款查询。
- 端到端测试覆盖扫码进入商户、下单支付、生成取餐号和后台核销完整流程。

## 16. 编码规范

- 必要代码注释必须使用中文，禁止新增英文解释性注释。
- 类、方法、字段、接口路径和标准技术标识使用英文命名。
- 注释只用于说明无法从代码直接看出的业务规则、并发控制和状态流转。
- 不为简单赋值、明显分支或重复代码含义增加无价值注释。
- 模块之间不得通过共享数据访问对象形成隐式耦合。

## 17. 演进原则

只有在真实容量、团队协作或独立发布需求出现后，才考虑拆分服务。优先候选是报表、支付集成或订单模块。拆分前保持模块接口清晰，不提前复制微服务基础设施。

多门店、套餐、人员排班、动态数据权限和消息队列均作为未来独立需求重新设计，不在第一阶段建立空壳模块。
