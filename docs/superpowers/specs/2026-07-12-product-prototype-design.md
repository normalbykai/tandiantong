# 摊点通第一阶段产品原型设计方案

## 1. 目标

为摊点通第一阶段建立可用于产品评审、UI 设计、AI 原型生成和前端开发的原型设计基线。本方案覆盖 C 端微信小程序和统一 B 端响应式管理后台；统一后台同时承载平台权限域与商户权限域，但后端接口和权限模型严格隔离。

本阶段只编写原型规格、流程图、关键线框图和 AI 生成提示词，不实现业务代码或可运行前端。

## 2. 交付文档

原型资料写入 `docs/product/`：

```text
docs/product/
├─ README.md
├─ 01-总体视觉与交互规范.md
├─ 02-C端小程序原型设计.md
├─ 03-B端统一后台原型设计.md
└─ 04-AI原型生成提示词.md
```

- `README.md`：阅读顺序、页面编号、角色与流程索引。
- `01-总体视觉与交互规范.md`：品牌、设计令牌、组件、状态和响应式规则。
- `02-C端小程序原型设计.md`：C 端页面规格、流程和关键线框图。
- `03-B端统一后台原型设计.md`：统一后台公共页面、平台权限域和商户权限域原型。
- `04-AI原型生成提示词.md`：全局提示词与逐页面提示词，可用于 Figma、MasterGo、墨刀及通用 AI 原型工具。

页面编号规则：

- `C-xx`：顾客端页面。
- `B-C-xx`：统一后台公共页面。
- `B-P-xx`：统一后台平台权限域页面。
- `B-T-xx`：统一后台商户权限域页面。

## 3. 视觉方向

### 3.1 C 端

C 端采用温暖烟火气风格，以暖橙和米白为基础，突出商品图片、服务信息和线下消费氛围。每个商户可配置 Logo、头图和主题色，系统限制主题色应用范围与文字对比度，确保可读性和状态色一致。

### 3.2 B 端

B 端采用清爽专业化风格，以蓝绿、白色和中性灰为主，强调信息密度、操作效率、待办事项和数据可读性。平台权限域与商户权限域共用基础设计系统，通过菜单、页面标题和身份标识区分。

### 3.3 基准画板

- C 端微信小程序：`375 × 812`。
- B 端桌面后台：`1440 × 900`。
- B 端移动后台：`390 × 844`。
- 移动端点击区域不得小于 `44 × 44 px`。
- 桌面表格在移动端转换为卡片或折叠分区，不进行等比例压缩。

## 4. C 端信息架构

底部导航固定为“首页、订单、我的”。订单页只展示商品订单，服务预约记录放在“我的预约”。

```text
首页
├─ 商户头部、营业状态、地址和公告
├─ 轮播图
├─ 商品优先 / 服务优先固定模板
├─ 商品分类与商品列表
├─ 服务预约推荐
└─ 购物车入口

订单
├─ 全部
├─ 待支付
├─ 待取餐
├─ 已完成
└─ 已取消 / 已退款

我的
├─ 顾客资料
├─ 我的预约
├─ 最近访问商户
└─ 服务说明
```

### 4.1 首页模板

商户可选择商品优先或服务优先模板。两种模板共用商户头部、轮播图、底部导航和基础组件，只调整商品区与服务区的顺序、数量和视觉权重，不支持自由装修。

首页支持 1 至 3 张 `16:9` 轮播图，可跳转商品详情、服务详情或仅展示。未配置轮播图时使用商户头图。首期不支持外链、弹窗广告和活动专区。

商品列表支持两种固定布局：

- 双列零售模式：顶部横向分类与双列商品卡片。
- 左侧分类点餐模式：左侧纵向分类与右侧单列商品列表。

### 4.2 商品购买流程

简单商品可在列表直接加入购物车；多规格或含加料商品打开 SKU 与加料选择面板；商品图片和名称可进入详情页。

```text
商户首页
→ 商品详情或规格加料面板
→ 当前商户购物车
→ 确认订单
→ 微信支付
→ 支付结果与取餐凭证
→ 到店核销
```

每个商户独立保存购物车，禁止跨商户合并结算。确认订单前服务端重新校验商品、SKU、加料项、价格、库存和营业状态。

顾客首次下单或预约时可授权或手动填写手机号，后续默认带入并允许本次修改。手机号用于联系和后四位辅助查询，不作为取餐凭证。

商户可配置是否启用预计取餐时间。启用后顾客选择立即取餐或配置时段；第一阶段不按取餐时段实施产能限制。

### 4.3 取餐凭证

取餐凭证以大号取餐号为视觉核心，二维码为安全核销凭证，同时展示预计取餐时间、商品摘要、门店地址、联系电话和退款入口。

取餐号仅用于识别，二维码令牌不可猜测。已核销后不再展示可用二维码，并显示核销时间。

### 4.4 服务预约流程

```text
首页服务入口
→ 服务详情
→ 近期日期与时段
→ 更多日期月历
→ 确认预约
→ 免费确认或微信支付
→ 预约凭证
→ 到店核销
```

近期日期采用横向日期栏，更多日期使用月历。时段展示剩余名额、价格和不可选原因。预约凭证以服务名称、预约日期和时间为视觉核心，二维码为辅助核销凭证。

## 5. 统一 B 端模型

统一 B 端使用一个前端工程、一个后台域名和一个登录页面。登录页提供“平台管理”和“商户管理”身份页签：

- 平台管理页签认证 `platform_user`，账号不绑定 `tenant_id`。
- 商户管理页签认证 `admin_user`，账号必须绑定 `tenant_id`。
- 第一阶段不允许一个账号同时跨两个权限域。
- 登录结果返回 `PLATFORM` 或 `TENANT`，前端加载对应菜单与工作台。
- 平台 API 使用 `/api/platform/v1/**`，商户 API 使用 `/api/admin/v1/**`，后端分别鉴权。

首个商户管理员和后续员工均使用邀请激活流程：验证手机号后自行设置密码，不传递明文初始密码。邀请凭证一次性使用并设置有效期。

## 6. 平台权限域

### 6.1 平台角色

- 超级管理员：平台全部权限，可创建和停用其他平台管理员。
- 商户运营：商户创建、资料维护、启用停用、管理员邀请和支付配置状态查看。
- 平台审计员：查看商户状态和平台日志，不允许修改。

首期平台角色预定义，不提供自定义平台角色。平台 RBAC 与租户 RBAC 完全独立。

### 6.2 平台页面

```text
平台工作台
├─ 商户状态统计
├─ 审核待办
├─ 支付配置待办
└─ 风险复核待办

商户管理
├─ 商户列表
├─ 新建商户向导
├─ 商户详情
├─ 启用 / 停用
├─ 支付配置
├─ 管理员邀请
└─ 小程序码管理

商户审核工作台
├─ 待启用
├─ 需修改
└─ 待复核

平台管理
├─ 平台管理员
└─ 平台操作日志
```

### 6.3 新建商户向导

新建商户采用分步向导并支持暂存草稿：

```text
主体信息
→ 门店资料
→ 首个管理员
→ 品牌设置
→ 汇总确认与创建
```

支付配置不阻塞创建。创建完成后商户处于“待启用”，可登录后台维护资料、商品、服务、员工和角色。支付未验证时禁止上架付费商品和付费服务，但允许配置和使用免费预约。

### 6.4 商户启用审核

商户状态流程为：

```text
草稿 → 待启用 → 需修改 → 待复核 → 已启用
```

启用前检查主体、门店、管理员激活、品牌资料、至少一项可用商品或免费服务，以及默认小程序码。需要付费业务时还必须验证微信支付配置。

平台端同时提供审核工作台和商户详情审核入口。驳回必须填写结构化问题清单，区分平台处理项与商户处理项。普通商户单人审核；主体、支付或违规风险升级超级管理员复核，高风险商户禁止创建人或最后修改人自审。

待启用期间提供仅租户管理员可访问的 C 端预览码。预览模式禁止创建真实订单、占用库存、占用预约容量或发起支付。

### 6.5 支付与小程序码

第一阶段每个租户绑定自己的微信支付商户号，资金直接进入商户账户。支付配置可后补，敏感字段加密存储并在页面脱敏显示。架构预留服务商模式，但首期不实现分账和内部结算。

商户启用后自动生成默认专属小程序码。入口使用随机标识，不直接暴露 `tenant_id`。重新生成时可选择旧码立即失效或设置过渡有效期。第一阶段不提供渠道码和来源统计。

## 7. 商户权限域

### 7.1 商户页面

```text

├─ 今日实收、订单、退款、预约
├─ 待核销、待履约、库存预警、异常支付
├─ 商品排行
└─ 预约时段使用率

商品与库存
├─ 分类
├─ 商品列表
├─ 商品分步编辑
├─ 入库、出库、盘点
└─ 库存流水

订单与退款
├─ 订单列表
├─ 订单详情
├─ 整单退款
└─ 异常支付

服务与预约
├─ 服务项目分步编辑
├─ 排期与容量
├─ 预约列表
└─ 预约日历

核销中心
├─ 扫码核销
├─ 取餐号查询
├─ 待核销列表
└─ 核销记录

经营数据
├─ 交易概览
├─ 商品分析
├─ 预约分析
└─ Excel 导出

员工与角色
├─ 员工账号
├─ 邀请激活
├─ 自定义角色
└─ 操作日志

商户设置
├─ 展示资料
├─ Logo、头图、主题色
├─ 首页模板与商品布局
├─ 营业规则
└─ 小程序码
```

### 7.2 响应式导航

桌面端采用左侧固定菜单和顶部工具栏；移动端切换为抽屉菜单。移动端优先展示核销和待办，经营图表后置。

### 7.3 商品编辑与库存

商品编辑采用五步流程：基础信息、规格与 SKU、加料配置、库存与上架、预览与保存。支持保存草稿和 C 端效果预览。复杂 SKU 编辑以桌面端为主。

初始库存可在商品创建时设置，后续通过入库、出库和盘点调整维护。系统自动记录订单锁定、支付扣减、取消释放和退款回补流水。首期不实现采购、供应商和门店调拨。

### 7.4 订单与核销

订单列表桌面端使用状态页签、筛选区和表格，移动端转换为卡片。支持订单号、取餐号、手机号后四位、支付状态、履约状态和时间筛选。

订单详情使用顶部状态摘要和“订单信息、支付退款、核销记录、操作日志”页签。移动端改为折叠分区。

核销中心同时提供扫码、取餐号或手机号后四位查询、待核销列表。核销前展示履约摘要并二次确认；重复核销显示原核销时间和操作人。

### 7.5 服务与预约

服务项目采用四步配置：基础信息、支付与预约规则、排期与容量、预览与上架。星期模板可批量生成排期，具体日期可调整或暂停。

预约管理同时提供列表和日历视图。桌面端支持月历和周历；移动端使用日程列表。两种视图共享筛选条件。

### 7.6 经营数据

经营数据支持按当前租户、日期和筛选条件导出 Excel。导出权限独立控制，敏感字段默认脱敏。大数据量使用后台任务生成临时文件并记录审计日志。首期不实现定时报表和邮件发送。

## 8. 商户状态与资料权限

“暂停营业”与“停用租户”分开：

- 暂停营业由商户管理员操作，暂停新订单和新预约，但允许登录、核销、退款和处理已有业务。
- 停用租户由平台管理员操作，禁止商户登录和业务操作，C 端显示暂停服务并保留历史数据。

平台维护商户主体、证件、租户状态、支付配置、首个管理员和平台备注。商户维护展示名称、Logo、头图、主题色、门店地址、联系电话、营业时间、首页模板和服务说明。

## 9. 关键线框图范围

文档必须绘制以下页面或页面组的线框图：

- 统一后台登录与邀请激活。
- 平台商户审核工作台。
- 新建商户分步向导。
- 商户详情与启用审核。
- 商户工作台桌面版和移动版。
- 商品分步编辑。
- 订单列表与订单详情。
- 预约日历与预约详情。
- 移动端核销中心。
- C 端商品优先首页和服务优先首页。
- SKU 与加料选择。
- 购物车与确认订单。
- 取餐凭证。
- 服务时段选择。
- 预约凭证。

普通设置和辅助页面使用完整页面规格与 AI 提示词描述，不要求逐页绘制线框图。

## 10. 页面规格要求

每个页面规格必须包含：

- 页面编号、名称、权限域、适用角色和入口。
- 页面目标与核心用户任务。
- 布局区域、字段、按钮、表格列或卡片内容。
- 默认、加载、空数据、失败、禁用、无权限和完成状态。
- 输入校验、二次确认、防重复提交和状态变化。
- 桌面、移动或小程序响应式要求。
- 与前后页面的跳转关系。
- 可直接交给 AI 原型工具的中文提示词。

## 11. 原型验收标准

- B 端公共、平台和商户权限域边界清楚，菜单隐藏不替代后端权限说明。
- 商户开通、审核、预览、支付配置和启用流程可完整走通。
- C 端商品购买、免费预约和付费预约流程可完整走通。
- 商品、订单、预约和核销状态在页面间表达一致。
- 桌面 B 端和移动核销场景均有明确布局。
- 每个关键页面包含异常、空数据和无权限状态。
- 设计令牌、组件名称和页面编号在四份文档中保持一致。
- AI 提示词明确画板尺寸、视觉风格、页面结构、真实示例内容和禁止事项。
- 不引入第一阶段范围之外的配送、套餐、员工排班、自由装修、平台分账或渠道营销能力。
